## Introduction

Modern AI coding assistants like GitHub Copilot Chat, Cursor, and Windsurf employ advanced **prompt shaping** and **context injection** techniques to interpret user requests with high semantic nuance. These systems retrieve relevant code snippets or documentation and pack them into the prompt to help large language models (LLMs) generate accurate answers. In our text adventure game’s command parser – implemented in pure Lua with no runtime neural inference – we face a similar challenge: interpreting **flexible, paraphrased user commands** (e.g. “*gimme the candle*” or “*I want to hold the candle*”) and mapping them to known `{verb, noun}` actions. Currently, our parser’s Tier 2 uses a **Jaccard token overlap** against a pre-generated list of \~4,579 phrasing variants (created at build-time via templates) to catch these non-exact matches. This yields about 20% additional command matches (for a total parser accuracy \~68%) but struggles with many novel phrasings and synonyms (especially the “very hard” and “extremely hard” cases in our benchmarks). [\[devblogs.m...rosoft.com\]](https://devblogs.microsoft.com/all-things-azure/github-copilot-chat-explained-the-life-of-a-prompt/), [\[explore.n1n.ai\]](https://explore.n1n.ai/blog/how-cursor-indexes-codebase-rag-pipeline-2026-01-27)

**Goal:** Explore how **Retrieval-Augmented Generation (RAG)** and context-management techniques from coding assistants can enhance our parser’s semantic matching without violating the **no runtime neural inference** constraint. We break this into six research areas:

1.  **Retrieval-Augmented Generation (RAG) for Command Parsing** – retrieval strategies beyond naive similarity (e.g. BM25, hybrid sparse+dense retrieval, GraphRAG).
2.  **Prompt Shaping & Context Packing in Constrained Environments** – how coding AIs rank and include relevant context, and how to optimally generate a compact set of command phrases.
3.  **Semantic Chunking & Token-Level Matching Beyond Jaccard** – advanced non-neural text similarity measures (weighted/soft Jaccard, token position, character n-grams).
4.  **Non-Neural Embedding Alternatives** – lightweight methods (LSH, random projection, static word vectors, feature hashing) to approximate semantic matching without heavy models.
5.  **Coding Assistant Prompt Architecture (Deep Dive)** – insights from GitHub Copilot, Cursor, etc. on how they structure and select context (e.g. local indexing, heuristics) that could inform our parser.
6.  **Practical Hybrid Approaches for Our Constraints** – combining the best of the above within our pure-Lua, offline-build setting, and evaluating effort vs. impact.

For each area, we summarize **seminal research**, **key findings**, and **applicability** to our scenario (short text commands, small vocabulary, offline processing). Finally, we present **3–5 concrete recommendations** (with estimated accuracy gains and implementation complexity) prioritized by effort-to-impact ratio.

## 1. Retrieval-Augmented Generation (RAG) for Command Parsing

**Seminal Papers & Systems:** RAG was introduced by Lewis *et al.* (2020) as a method to boost knowledge-intensive tasks by retrieving relevant text from a large corpus and providing it to a generator model. Their **Retrieval-Augmented Generation** (NeurIPS 2020) combined a parametric seq2seq model with a non-parametric memory (a dense vector index of Wikipedia) and set new state-of-the-art accuracy on open-domain QA tasks. Key related developments include **Dense Passage Retrieval (DPR)** for efficient question retrieval (Karpukhin *et al.*, 2020) and **Facebook’s RAG** models, which improved factual QA accuracy by \~8–15 points over knowledge-unaugmented baselines (exact gains vary by dataset). More recently, Microsoft Research’s **GraphRAG** (Han *et al.*, 2024/25) proposes a hierarchical retrieval using knowledge graphs instead of flat text; it achieved higher answer diversity and accuracy on complex queries (like multi-hop questions) compared to standard RAG. Additionally, **hybrid retrieval** methods such as **ColBERT** (Khattab & Zaharia, SIGIR 2020) and **SPLADE** (Formal *et al*., SIGIR 2021) incorporate sparse lexical matching signals into dense vector search, combining the precision of term matching (e.g. BM25) with the semantic reach of embeddings. [\[arxiv.org\]](https://arxiv.org/abs/2005.11401) [\[ibm.com\]](https://www.ibm.com/think/topics/graphrag), [\[ibm.com\]](https://www.ibm.com/think/topics/graphrag) [\[explore.n1n.ai\]](https://explore.n1n.ai/blog/how-cursor-indexes-codebase-rag-pipeline-2026-01-27), [\[marktechpost.com\]](https://www.marktechpost.com/2026/03/22/how-bm25-and-rag-retrieve-information-differently/)

**Key Findings:** These works show that **pure semantic similarity (dense vectors)** and **pure lexical similarity (keyword overlap)** have complementary strengths and weaknesses. Lexical methods like **BM25** (Okapi) excel at precision for exact term matches and don’t require heavy computation. BM25 uses term frequency, document length normalization, and inverse document frequency (IDF) to rank results. It’s long been the backbone of search engines (Lucene/Elasticsearch) due to its strong performance and efficiency. However, lexical matching fails when user queries use synonyms or paraphrases not present in the indexed text (e.g. “*grab candle*” vs indexed “*take candle*”). Dense **semantic retrieval** overcomes this by matching the *meaning* of queries and documents in vector space – e.g. finding “heart attack” ≈ “myocardial infarction” despite no shared words. The trade-off: semantic search requires powerful models at query time, whereas BM25 is lightweight and deterministic. State-of-the-art *production* RAG systems therefore use **hybrid retrieval**: retrieve candidates with efficient lexical methods and re-rank or augment with semantic similarity. For instance, Cursor’s pipeline first uses an embedding index to find concept-related code *and* a keyword index for exact symbol matches, merging results. This yields high recall and precision: unique identifiers or literal phrases are caught by lexical search, while semantic search covers rephrased or conceptually related matches. Microsoft’s **GraphRAG** goes further: it performs a **two-stage retrieval** by first using a knowledge graph to narrow down relevant documents (as a classifier by entity/relations), then does vector search within those documents. This hierarchical approach improved answer recall and “comprehensiveness” by \~10–20% on complex queries, all while *reducing* the amount of text needed in the prompt by up to 97% in experiments (via graph-based context summarization). [\[marktechpost.com\]](https://www.marktechpost.com/2026/03/22/how-bm25-and-rag-retrieve-information-differently/) [\[marktechpost.com\]](https://www.marktechpost.com/2026/03/22/how-bm25-and-rag-retrieve-information-differently/), [\[marktechpost.com\]](https://www.marktechpost.com/2026/03/22/how-bm25-and-rag-retrieve-information-differently/) [\[explore.n1n.ai\]](https://explore.n1n.ai/blog/how-cursor-indexes-codebase-rag-pipeline-2026-01-27) [\[towardsdat...cience.com\]](https://towardsdatascience.com/graphrag-in-practice-how-to-build-cost-efficient-high-recall-retrieval-systems/), [\[towardsdat...cience.com\]](https://towardsdatascience.com/graphrag-in-practice-how-to-build-cost-efficient-high-recall-retrieval-systems/) [\[ibm.com\]](https://www.ibm.com/think/topics/graphrag), [\[ibm.com\]](https://www.ibm.com/think/topics/graphrag)

**Direct Applicability:** For our parser, RAG’s insight suggests **using smarter retrieval** on the player’s input *before* attempting a direct match. In practice, we could integrate a **non-neural retrieval step** that goes beyond raw Jaccard overlap. For example, implementing a **BM25 scorer** for our phrase index is feasible in pure Lua (using precomputed IDF weights and term frequencies at build time) – essentially treating each phrase variant as a “document” and the user’s command as a query. Unlike Jaccard (which equally weights all tokens and ignores term rarity or frequency), BM25 would boost matches on rarer, informative words (e.g. object names) and handle multi-word inputs more gracefully. This could immediately improve matching when the user input contains extra adjectives or synonyms: common filler words contribute less to BM25 score than meaningful words. Another approach is **hybrid retrieval without neural models**: for instance, use a **boolean or fuzzy keyword filter first** (to gather candidate phrases that share at least one important token with the query), then apply a more semantic similarity measure on those candidates. This is akin to how some RAG systems first retrieve by keywords and then re-rank by embedding similarity – except we’d replace the second stage with a non-neural “semantic” metric (see Section 3 on soft matching). We could also consider a **two-stage cascade** analogous to GraphRAG’s hierarchical strategy: e.g., first determine the likely verb (maybe via a simple classifier or lookup) to narrow the search space to relevant verb-specific phrases, then match the noun phrase within that subset. Our current Tier 1 already does some of this by exact verb matching, but we could strengthen it (for example, using a small **verb classifier or heuristic** to catch verbs even when phrased indirectly). This two-stage approach could cut down false positives and speed up search by not comparing against all 4,579 phrase variants every time. [\[marktechpost.com\]](https://www.marktechpost.com/2026/03/22/how-bm25-and-rag-retrieve-information-differently/) [\[explore.n1n.ai\]](https://explore.n1n.ai/blog/how-cursor-indexes-codebase-rag-pipeline-2026-01-27)

**Non-Neural Methods beyond Jaccard:** The literature offers several non-neural similarity metrics that could surpass straightforward Jaccard overlap for short text:

*   **BM25 / TF-IDF:** Widely used in IR, these assign weights to terms based on frequency and rarity, improving relevance ranking for short queries. They are **completely runnable in Lua** (just algebra on term counts and precomputed IDF constants). Using BM25 on our data would require indexing each phrase’s token frequencies and computing an IDF for each token across the corpus at build time. At runtime, scoring 4.6k phrases with BM25 (a sum of a few terms’ scores per phrase) is still fast (should be on the order of milliseconds). We anticipate BM25 could improve Tier 2’s hit rate by making it less sensitive to extraneous words: e.g., an input “*get the big candle from shelf*” would still score the *{get,candle}* phrase highly because *“candle”* is a rare token with high IDF, whereas Jaccard might down-rank it due to extra words. [\[marktechpost.com\]](https://www.marktechpost.com/2026/03/22/how-bm25-and-rag-retrieve-information-differently/)
*   **Alternative Lexical Metrics:** Measures like **Dice coefficient** or **cosine similarity on TF vectors** are similar to Jaccard but include term weights. Prior studies (e.g., Robertson & Zaragoza 2009) detail that these TF-IDF weighted cosine/BM25 approaches generally outperform unweighted Jaccard in document retrieval. For our short texts, the improvement might be smaller but still meaningful (perhaps a few percentage points in accuracy). We can experiment with a simple **TF cosine similarity** (treat each phrase as a vector of term weights) or **Dice coefficient** to see if it improves ranking of correct commands. [\[marktechpost.com\]](https://www.marktechpost.com/2026/03/22/how-bm25-and-rag-retrieve-information-differently/), [\[marktechpost.com\]](https://www.marktechpost.com/2026/03/22/how-bm25-and-rag-retrieve-information-differently/)
*   **Hybrid Sparse+Dense Retrieval:** True dense vector matching (like using our GTE-tiny embeddings at runtime) is off the table due to the pure-Lua constraint. However, we can simulate aspects of dense retrieval **offline**. One idea is to use our archived GTE-tiny embeddings of the 4,579 phrases to assist a lexical method. For example, we could cluster the embedding space (e.g., via *k*-means or hierarchical clustering during build) to group semantically similar phrases. At runtime, given an input, we could do a lightweight similarity estimation to those cluster centroids to choose a likely cluster of candidate phrases, then only Jaccard-match within that cluster. This is analogous to an Annoy or FAISS (approximate nearest neighbor) index, but we could implement a basic version with precomputed cluster centers (the input would have to be assigned to a cluster by a heuristic – perhaps by comparing the input’s keywords to cluster keywords or to exemplar phrases). While less precise than true vector search, this could prune irrelevant candidates. However, the benefit might be limited given our small index size (4.6k is manageable with brute force). A simpler hybrid approach is: **if Jaccard similarity is below a threshold**, the parser could fall back to more permissive matching strategies (like the soft matching in Section 3) or even a second-pass where *stop-words are removed entirely* and we just look for any overlap of content words. This mimics a “sparse + dense” fallback: first pass requires some exact token overlap (for precision), second pass allows looser matches if nothing strong was found (for recall).

In summary, RAG research emphasizes retrieving and supplying the *right context* to the language model. For our parser, this translates to retrieving the right candidate command(s) from our known corpus of valid actions, using smarter algorithms than a flat Jaccard scan. Approaches like BM25 and hybrid retrieval can be **adapted in pure Lua**, since they rely on classic IR math (no neural nets at runtime) and data structures (inverted indexes, precomputed weights). They address the “query encoding” problem (player phrasing vs stored phrasing mismatch) by bridging the gap with lexical and semantic cues. We should leverage these ideas to catch paraphrases without a full transformer: e.g., **BM25 scoring combined with offline semantic clustering or synonym matching**.

**Estimated Impact:** Introducing BM25 or improved retrieval is likely one of the **highest-impact, low-risk changes**. Prior research in information retrieval shows BM25 often beats simple overlap metrics. While no direct benchmark exists for text commands, we can extrapolate from code search and short-text IR tasks. For example, a 2024 study integrating BM25 with Jaccard in a search engine saw notable improvements in retrieval precision. We might expect a modest boost in Tier 2 hit rate – perhaps on the order of 5–15% relative improvement in catching paraphrased commands, especially those with extra filler words or multiple tokens. The hybrid two-pass approach could rescue additional cases (like “gimme the…”) that are currently missed, possibly raising overall command resolution accuracy from \~68% to 75–80%. These are estimates; we would need to measure on our 60-test benchmark or gather new player input data for validation.

**Implementation Complexity:** **Effort: low-to-moderate.** Writing a BM25 scorer in Lua involves parsing the input into tokens (we already do this), computing a score for each candidate phrase, and returning the max. The math (log IDF, term frequencies) is straightforward algebra. We’d need to precompute and store IDF values for each term in our 4.6k phrase corpus at build time – trivial size. A full inverted index (mapping each token to phrase list) would make it even faster by only scoring phrases containing query tokens, though given only \~5k candidates, brute force is <100ms in worst-case anyway. Managing a hybrid pipeline (lexical + semantic) might need a bit more code (two phases of scoring and merging results) but still pure Lua logic. **Risk:** minimal, since these methods are well-understood and easily debuggable (we can test them offline on our example inputs). We must tune BM25 parameters (`k1`, `b`) for short texts (likely `k1≈1.2`, `b≈0.75` are standard). GraphRAG’s full approach (building a knowledge graph of game concepts) would be overkill for our scale, but the idea of *structured retrieval* could inspire future enhancements (see Section 6). ## 2. Prompt Shaping & Context Packing in Constrained Environments

**Seminal Papers/Examples:** AI coding assistants rely on selecting the right context to include in the prompt due to limited model input length. Key strategies are discussed in Microsoft’s *“GitHub Copilot Chat: The Life of a Prompt”* (Banerjee, 2025), which outlines how the VS Code extension maintains a **local index of the workspace**, finds relevant code snippets (e.g. functions related to the user’s question), and packages only the most relevant ones into the prompt (to avoid exceeding token limits). Research on **prompt/context optimization** often overlaps with query expansion and active learning: e.g., Rocchio’s algorithm (1971) and relevance feedback techniques dynamically expand or reweight query terms to improve retrieval. More recent work on **active learning** for IR (e.g. Zhai & Lafferty 2001 on feedback and term reweighting) and corpus construction suggests selecting examples or terms that maximize coverage of semantic space. The **Set Covering Machine** (SCM by Marchand & Shawe-Taylor, 2002) and subsequent work formulated the problem of choosing a minimal set of features (or examples) that “cover” the concept space – this is analogous to picking a minimal set of phrasing templates that cover all likely player utterances. In general, finding an *optimal* set of prompts or index entries is NP-hard (it parallels the **set cover problem**, which is NP-complete to decide and NP-hard to optimize). Thus, practical approaches use heuristics: e.g., greedy algorithms for set cover achieve an approximation factor $H(n)$ (Harmonic number), meaning they cover most cases with a limited increase in set size. [\[devblogs.m...rosoft.com\]](https://devblogs.microsoft.com/all-things-azure/github-copilot-chat-explained-the-life-of-a-prompt/), [\[devblogs.m...rosoft.com\]](https://devblogs.microsoft.com/all-things-azure/github-copilot-chat-explained-the-life-of-a-prompt/) [\[en.wikipedia.org\]](https://en.wikipedia.org/wiki/Set_cover_problem)

**Key Findings:** In coding assistive tools, **ranking heuristics** for context selection are crucial. Copilot and Cursor use several signals:

*   **Recency & Focus:** Files or functions the user edited or viewed recently are boosted in relevance (under the assumption that the question likely relates to current work). For instance, Cursor’s docs mention you can configure priority for certain folders or recent files. [\[explore.n1n.ai\]](https://explore.n1n.ai/blog/how-cursor-indexes-codebase-rag-pipeline-2026-01-27)
*   **Similarity scoring:** They typically maintain vector embeddings of code/comments and use semantic search to find code related to the query. But even without neural models, one could use simpler metrics (like overlapping identifiers or function names) to guess relevance. The Copilot extension’s local indexing likely involves scanning for symbol names or keywords from the query in the codebase. [\[devblogs.m...rosoft.com\]](https://devblogs.microsoft.com/all-things-azure/github-copilot-chat-explained-the-life-of-a-prompt/)
*   **Context size limits:** Context packing algorithms try to include as much relevant info as possible until the token limit is reached, often preferring a larger number of small relevant snippets over one large file. This is analogous to our parser’s need to include enough phrase variants to cover language variety without bloating the index. Techniques like **Maximal Marginal Relevance (MMR)** (Carbonell & Goldstein, 1998) reorder retrieved items to maximize relevance *and novelty*, ensuring a diverse set of contexts (preventing redundant info from consuming space).

For our parser, **prompt/context packing** equates to **how we design and maintain the phrase variant index** and how we might dynamically adjust it. Key insights:

*   *Optimal Phrase Variant Generation:* Our current method enumerates templates for each verb–noun pair (like “get X”, “pick up X”, “take X”, etc.). This covers many synonyms by design, but we saw novel phrasings like “gimme \[object]” or “I want to \[verb] \[object]” falling through. Research on **query expansion** can guide us to generate better variants. For example, using a thesaurus or WordNet to find synonyms for verbs and common phrasal structures can systematically expand the coverage. *Lu et al.* (2015) demonstrated that adding WordNet synonyms to search queries improved code search precision by \~5% and recall by \~8%. We could apply this offline: for each verb and noun in our game, look up synonyms (and maybe common slang) and add those to the phrase list. Caution is needed to avoid irrelevant synonyms (WordNet can suggest archaic or contextually wrong terms). Lu *et al.* addressed this by matching part-of-speech and filtering out unlikely senses. In our domain, we might hand-pick synonyms for key verbs (e.g., *“take”*: *get, grab, pick up, snatch, seize*; *“attack”*: *hit, strike, whack*; etc.) to ensure they fit the context. [\[risame.github.io\]](https://risame.github.io/sun/query.pdf) [\[risame.github.io\]](https://risame.github.io/sun/query.pdf), [\[risame.github.io\]](https://risame.github.io/sun/query.pdf)
*   *Active Learning for New Phrases:* If we had a log of real player inputs, we could apply **relevance feedback**: analyze which inputs were not matched and add new aliases for them. In absence of user data (for a new game), another approach is to use an LLM at *build time* to generate variations. For example, we could prompt GPT-4 in a script to paraphrase a set of command phrases (e.g., “other ways someone might say ‘pick up the lamp’?”). This is similar to how some AI-assisted testing tools generate user utterances variations. While this uses a neural model, it’s offline (during development, not in the game runtime) so it doesn’t violate the no-runtime-ML rule. This could yield creative phrasings (like “scoop up the lamp”, “grab hold of the lamp”) that our templated approach missed, effectively simulating user input. Each new candidate could then be verified (perhaps via the embedding similarity or developer review) before inclusion in the index.
*   *Query Expansion at Runtime:* If adding every possible variant makes the index too large, another tactic borrowed from IR is **on-the-fly query expansion**. That is, when a user input arrives, we could automatically expand it with likely synonyms *before* matching. For instance, if input contains “gimme”, we could substitute or append “give me”/“get” before matching. This could be powered by a **precomputed lexicon** of known synonyms/slang to canonical verbs. For example, maintain a dictionary: {"gimme"⇒"give","pickup" (noun)⇒"take","opal"⇒"gem"} and so on. In information retrieval, techniques like **Rocchio’s algorithm** adjust a query vector towards relevant term vectors based on known good results, but in our simpler context, a manual synonym table might suffice. This is essentially a lightweight form of prompt shaping: reshape the user’s command into a more standard form that our index recognizes.
*   *Minimal Context & Overlap:* There is a trade-off between covering more language variety and keeping the phrase index small (to limit memory and speed overhead). The set cover perspective indicates that finding the absolute minimal set of phrases that cover all paraphrases is intractable to solve optimally. However, we might use heuristics to trim the fat from our 4,579 phrases. For example, if two phrases are very similar, one might be redundant. We could measure the coverage of each phrase in terms of capturing possible user utterances (perhaps by using our embedding space: if two template phrases have extremely similar GTE vectors, one might be enough). This is analogous to **submodular optimization for summary coverage** in NLP: choose a set of sentences (phrases) that maximize coverage of meanings. While an exact solution is hard, even a greedy selection could reduce the index size without sacrificing recall much. That said, since 362 KB of JSON is not too large for our use case, optimization here is a lower priority than improving accuracy. [\[en.wikipedia.org\]](https://en.wikipedia.org/wiki/Set_cover_problem)

**Direct Applicability:** The main takeaways are to **simulate what the coding assistants do, but at design-time**:

*   Build a *smarter phrase index* using external knowledge (synonym lexicons or even LLMs for generating paraphrases) to achieve broader coverage of player expressions. This is effectively doing **prompt engineering offline** – preparing the parser’s “knowledge base” of possible phrasings in a way that anticipates user’s linguistic variety.
*   Use simple **heuristics to prioritize context** at runtime. For example, if we know the player is in a certain location or recently mentioned an object, we might boost phrase variants that involve items in the current room (similar to how Copilot boosts current file code context) – this can help disambiguate pronouns or implicit nouns. Our Tier 4 already uses recent context for pronoun resolution, but we could extend this idea: maintain a short list of recently referenced nouns and, if the user input is vague (e.g. “take it” or “use key” when multiple keys exist), prefer the one mentioned most recently or present in the room. While not exactly drawn from coding assistants, it’s analogous to how they use the “current context” to interpret requests.

**Estimated Improvement:** By expanding our phrase set with synonym-based variants, we can likely close a significant portion of the 30% gap in the “novel phrasing” category of our failures. Lu *et al.* (2015) saw 5–8% gains in code search using WordNet expansion; in our controlled environment, perhaps **5–10% overall parser accuracy gain** is achievable via offline synonym augmentation. Query expansion at runtime (e.g. replacing slang with canonical terms) would directly fix some “unrecognized” cases – likely eliminating many of the 80% of “very hard” cases that Jaccard currently misses (those often involve slang or extra verbiage). Active learning or LLM-generated expansions could ensure that as players discover new phrasings, the parser evolves to handle them, preventing accuracy from stagnating. These improvements target exactly our weakest category (novel phrasing, currently 20% success) – we might raise that to 50% or more with a thorough expansion strategy, bringing overall accuracy into the mid-70s or higher. [\[risame.github.io\]](https://risame.github.io/sun/query.pdf)

**Implementation Complexity:** **Effort: moderate (mostly one-time build-time effort)**. Building a richer phrase index with synonyms requires either integration of a source like WordNet (which we can do via a Python script using NLTK’s WordNet before packaging the game) or manually curating lists of synonyms/slang for each verb. The former is more systematic (but needs careful curation to avoid nonsense phrases); the latter is more labor-intensive but ensures quality. Using an LLM for paraphrase generation at build-time is relatively easy (just a script calling an API) though verifying the outputs will need some manual review to avoid adding bizarre phrases. On the runtime side, implementing a simple query expansion (word replacement) using a predefined dictionary is straightforward in Lua. Maintaining context (recent nouns/verbs) for disambiguation or prioritization is also feasible with small changes to how the parser stores state. None of these require heavy computation during gameplay – they mostly shift the burden to design time.

## 3. Semantic Chunking & Token-Level Matching Beyond Jaccard

**Seminal Research:** *Chunking* in text processing refers to breaking text into meaningful units. In code RAG, Cursor’s approach is to chunk by **semantic units** (functions, classes) rather than arbitrary lengths, ensuring each chunk has standalone meaning. In text commands, our analog is simpler: each command is already a short “chunk,” but we might consider multi-word units (like phrasal verbs or compound nouns) as single semantic units. For similarity measures, classic research by Zhao *et al.* (ICDM 2014) and others have surveyed short text similarity techniques beyond exact token overlap. One important approach is the **soft cosine similarity** (Sidorov *et al*., 2014) which incorporates pairwise token similarity into the calculation. Instead of treating tokens as completely distinct dimensions, soft cosine uses a similarity matrix between features – e.g., it can recognize that *“grab”* is 0.8 similar to *“get”*, so an input like “grab lamp” and an index phrase “get lamp” would have a high similarity even with no exact token match. Another variant is **soft Jaccard** (often called soft overlap), which extends Jaccard by counting *similar* words as partial overlaps. For instance, *“sword”* and *“blade”* might count for 0.7 overlap if the words are related. Other non-neural methods include: [\[towardsdat...cience.com\]](https://towardsdatascience.com/how-cursor-actually-indexes-your-codebase/)

*   **Character N-gram Overlap:** Measures like **Dice coefficient** or **Jaro-Winkler** on character *n*-grams provide robustness to minor spelling differences or inflections. These have been used in record linkage and short text comparison (e.g., matching “armor” vs “armour” or handling typos). Our Tier 5 already uses Levenshtein distance for typos, which is similar in spirit. We could potentially incorporate character bi-gram overlap as another signal (it’s computationally cheap for short strings). For example, *“examine”* vs *“inspect”* share few words, but at the character level there’s some overlap (“in…ect”); still, semantic relation is more effective than chargrams for different words entirely.
*   **POS Tagging & Matching Structure:** Some research (e.g., on question answering systems) suggests leveraging the grammatical structure of input. For our domain, one idea is to parse the command lightly (identify the verb and object). We already attempt this by splitting on spaces and removing articles. We could enhance it by lemmatization (so *“looking”* → *“look”*, *“pick up”* → *“pick\_up”* as one verb unit). Using a small morphological dictionary or rules (e.g., common verb conjugations, plural -> singular nouns) would effectively increase overlap for things like “doors” vs “door”, “go north” vs “going north”. This is a well-trodden path in IR; most search engines strip suffixes using stemmers (Porter, 1980) or lemmatization to deal with morphological variants.
*   **Term Weighting and Ordering:** Our Jaccard metric treats all tokens equally and ignores word order. In commands, word order is usually less crucial (since they’re short and often telegraphic), but *some* preference for matches that preserve contiguous word sequences could help. For instance, an input “turn on the lamp” vs a phrase “turn the lamp on” – both contain “turn”, “on”, “lamp”, but the relationships differ. However, most adventure parsers ignore grammar (they treat “turn lamp on” equivalently). So ordering might not be critical. What could help more is weighting certain tokens more. *Verbs vs nouns:* Possibly weight the verb token higher, because a correct verb match is necessary, whereas an extra adjective on the noun is less critical. E.g., in “hit the ugly troll”, the verb “hit” and noun “troll” are key; the adjective “ugly” can be discounted. We could implement a simple weighting: in similarity score, give full credit to overlapping verbs and maybe a half-weight to overlapping adjectives or other context words. This idea is supported by our domain knowledge and can be informed by the part-of-speech tagging of the phrase (which we could do at build time by a small POS tagger or simply by pattern: if a word matches one of our 48 verbs list, treat it as verb, etc.). Weighted Jaccard indexes exist in literature (Chaudhuri *et al.*, VLDB 2006, introduced a weighted Jaccard for set similarity joins). In our case, a custom weight scheme (like verb = 2x, noun = 1.5x, others = 1x) could be tuned by trying a few values on known tough examples.

**Key Findings:** Non-neural similarity methods can close some gap to neural embeddings by incorporating domain knowledge:

*   **Soft Cosine / Soft Jaccard:** Sidorov *et al.* (2014) showed that introducing a word similarity matrix into cosine similarity improves retrieval and clustering of short texts where exact word overlap is low. For example, if *sim(word1, word2)* is available (from a thesaurus or embeddings), soft cosine will count word1 and word2 as partially matching. This was found to notably improve tasks like question answering and duplicate question detection on forums, where phrasing varies but meaning is the same.
*   **Character-Level Similarity:** Earlier IR work (e.g., Cohen *et al.*, 2003 on “soft TF-IDF” and fuzzy matching) demonstrated that combining character edit distance with token matching helps catch spelling variations and typos. Our Tier 5 already uses Levenshtein distance (which is a dynamic programming edit-distance algorithm), covering this aspect. We might explore if a simpler and faster n-gram overlap could replace or augment Levenshtein. However, since our inputs and commands are short, the performance impact is negligible; correctness is more important.
*   **BM25 vs Jaccard on Short Texts:** While BM25 is known to outperform Jaccard in longer documents, it can aid short text matching too. A 2024 study by Bronda-Ecraela & Doctora integrated BM25 with Jaccard for thesis title search and reported better mean average precision than Jaccard alone. The benefit arises from BM25’s consideration of global term rarity and per-query term importance, which even in a short query/candidate setting can help distinguish signal words from noise.

**Direct Applicability:** There are two especially promising techniques for our parser: **soft matching via word similarity** and **weighted lexical scoring**.

*   *Word Similarity Matrix (Embedding-based soft match):* Using our GTE-tiny model’s archive, we can compute an offline **word-to-word similarity matrix** for the limited vocabulary of our game (which appears to be on the order of a few hundred unique words). For instance, we calculate cosine similarity between the embedding of every pair of words (or stems) in our verbs/nouns list. This yields a matrix where, say, *sim(“get”, “grab”)* = 0.95, *sim(“take”, “grab”)* = 0.90, *sim(“candle”, “lamp”)* = 0.88, etc. At runtime, we can use this to implement a *soft Jaccard*: when comparing input tokens to a phrase’s tokens, count a partial overlap score for similar words. For example, if input = “grab candle” and candidate = “get candle”, exact Jaccard would be 1/3 = 0.33 (the tokens sets might be {grab,candle} vs {get,candle} intersection = {candle}). But if we consider *grab ≈ get* (0.95 similarity), we can effectively raise that score near 1.0, correctly identifying it as a strong match. This approach was essentially proposed by Sidorov *et al.* and others for short text comparison problems, and can be implemented without a neural net at runtime. We just do additional lookups in our similarity matrix. We should choose a threshold for what constitutes a “match.” For instance, treat sim ≥0.8 as a full match, or multiply the Jaccard overlap count by the sim value for non-exact matches. The matrix could also encode morphological relations (so “take” vs “taking” = 1.0). The matrix creation is $O(N^2)$ in vocabulary size – with N\~200, that’s \~40k pairs, which is fine to compute offline. The memory footprint would be N^2 floats; if N=200 and each sim is 4 bytes, that’s \~40k\*4 = 160kB, trivial. We could even quantize to 1 byte per similarity (0–255) to save space, or store only above-threshold pairs in a sparse structure.
*   *Weighted or Filtered Overlap:* Another simpler method is to manually designate certain words to ignore or down-weight. We already drop articles (“the”, “a”) in preprocessing. We might extend the stop word list to include very common verbs that don’t carry action meaning by themselves in our domain. For example, players might type “please *let me* take the sword” – the words “please” and “let me” could be safely dropped. We can compile a list of polite filler phrases or common extraneous verbiage to strip out (some parsers do this). Additionally, as noted, give different weights to different parts of speech: ensure the primary verb match is valued highly. We could implement a scoring function where the presence of the correct verb in the input adds a big bonus. This ensures that an input missing the right verb (even if it shares some nouns) doesn’t erroneously match a wrong command (one of our observed failure modes was “hold \[object]” vs “get \[object]” confusion; if “hold” maps to “get” with some similarity but we know “hold” is not a canonical verb in our game, we should probably treat “hold” as an attempt at “take” but that should be handled via synonyms rather than matching “hold” to a different verb). So, perhaps better: build a map of out-of-vocabulary verbs to the closest known verb (similar to query expansion). “Hold” → “get”, “stash” → “drop”, etc. Some of this can be done by the word similarity matrix as well (finding nearest neighbor verb).
*   *Phrase Structure Awareness:* We could consider bigram overlaps to incorporate some word order (the so-called **Skip-Bigram** similarity). For example, if a player writes “turn quickly on lamp”, bigram matching might catch “turn on” as a phrase. But the benefit here is limited and might complicate things. Our commands are usually 2-3 words, so bigram matching might be overly strict. Instead, focusing on the presence of the key components (verb, noun, maybe preposition if needed) is enough.

**Estimated Improvement:** Soft matching using word similarities is likely to yield a **substantial accuracy boost on the “very hard” phrasings**, where completely new words or phrasings are used. By leveraging the semantic knowledge imbued in GTE-tiny (or even a smaller embedding source), we could correctly map many previously missed commands. For instance, “use the sword on the troll” might not match “attack troll” by Jaccard, but if *use* is semantically close to *attack* in some contexts, we’d catch it. It’s hard to quantify expected gain without testing, but we might reclaim a large fraction of that 80% failure on novel phrases – possibly improving it to 50–60% success. That alone could raise overall parser accuracy by \~10%. Weighted scoring and better normalization (like BM25) will contribute smaller increments – maybe a few percentage points – by fixing edge cases (e.g., inputs with extra fluff or multi-word objects). The combination of these improvements could realistically push our Tier 2 success from \~20% coverage to well above 30%, and total parser accuracy into the mid-70s or higher (from 68%).

**Implementation Complexity:** **Effort: moderate.** We will need to integrate additional data and algorithms:

*   Building and storing a word similarity matrix from GTE-tiny embeddings can be done in Python. We already have the embeddings; it’s a matter of computing cosine for each pair of words. We must then include this data in the game (perhaps as a Lua table or a compressed structure). Lookup at runtime is $O(1)$ per word pair, so overhead is small. The coding in Lua to use the matrix in similarity calculations is straightforward. The main complexity is tuning how to integrate these similarities into the matching score (designing a formula that translates a set of token similarities into a single score – soft Jaccard or soft cosine provide a guide). [\[risame.github.io\]](https://risame.github.io/sun/query.pdf)
*   Weighted Jaccard or BM25 scoring also require some tweak to our scoring function, but this is again just adjusting how we count matches. We might implement a unified scoring function that, for each candidate phrase, computes:
    $$\text{score} = \sum\_{t \in \text{input\_tokens}} \sum\_{u \in \text{phrase\_tokens}} w(t, u)$$
    where $w(t,u)$ is 1 if tokens are identical, or some fraction if they’re similar (and maybe boosted if they’re verbs, etc.). This is a generalized soft overlap. We’d then choose the candidate with max score.
*   We should be careful with performance: a double loop over input and phrase tokens for 4.6k phrases. Worst-case input maybe \~5 tokens and phrase \~5 tokens, so \~25 comparisons per phrase, \~115k comparisons total – absolutely fine in <0.1s. If we add more phrases or tokens, it could grow, but likely okay up to some tens of thousands of comparisons (Lua can handle that per frame).
*   One caveat: We must ensure these new techniques don’t produce **false positives** that reduce precision. If we allow too fuzzy a match, the parser might erroneously match the wrong command. We’ll need to test and potentially set similarity thresholds (e.g., require the soft score to be above a certain value to accept). The advantage is we can calibrate this offline using our test set and any additional sample commands we can think of. The disambiguation and context from Tier 3–5 (idioms, pronouns, etc.) will still apply.

## 4. Non-Neural Embedding Alternatives

**Seminal Concepts:** When neural models are not feasible at runtime, computer scientists have long employed algorithmic alternatives for similarity search and embedding approximation:

*   **Locality-Sensitive Hashing (LSH):** Indyk & Motwani (STOC 1998) introduced LSH as a way to hash items so that similar items have a high probability of colliding in the same bucket. For text, one can use character *n*-gram shingles and hash them (this was used by Broder (1997) to detect near-duplicate web pages using MinHash signatures of shingle sets). LSH allows *approximate nearest neighbor* search in sub-linear time by comparing only items that land in the same bucket. For our scale (4k phrases), LSH is unnecessary for performance, but conceptually we could precompute MinHash signatures for each phrase (to approximate Jaccard similarity) and then at runtime hash the input and only compare to phrases with at least one shared bucket. This might speed up matching if the phrase list grows large, but it won’t fundamentally improve accuracy. So LSH is more of a performance optimization than an accuracy booster.
*   **Random Projections & Semantic Hashing:** The Johnson-Lindenstrauss lemma (1984) guarantees that high-dimensional vectors can be projected into a lower-dimensional space with minimal distance distortion. In 2008–2009, researchers (e.g., Salakhutdinov & Hinton’s semantic hashing) explored using *autoencoders* or random projections to map text documents to binary codes that preserve similarity. If we were to try to use our 384-d GTE vectors at runtime without the 17M model, one idea is to perform a heavy dimensionality reduction offline. For example, we could apply PCA or random projection to reduce 384 dimensions to, say, 32 dimensions, and then include those 32-d vectors in our game. 32 dims per phrase (float32) for 4,579 phrases is \~ 4,579×32×4 bytes ≈ 586 KB – possibly acceptable. Then we’d need to compute a *query vector* for the input in the same space **without a neural model**. This could be done by a simpler encoding scheme: perhaps summing known word vectors (if we also pre-store a small embedding for each word). We tried a naive version of this (bag-of-words average of static word vectors) and got \~45% accuracy (worse than Jaccard’s 68%). But maybe a smarter linear projection or a small model could do better. There is research on **“deep averaging networks”** or **knowledge distillation** that might help: e.g., training a shallow network to mimic the output of a BERT on known sentences (Turc *et al.*, 2019; Hinton *et al.*, 2015 on knowledge distillation). With 4,579 training examples (our phrases) and their corresponding GTE embeddings, we could train a small feed-forward network to map a simple representation of the input (like a 200-d bag-of-vocab vector or a 2-layer LSTM) to the embedding space. If successful, this would let us encode new inputs similarly at runtime with the small model. However, training such a model is complex and the result might still be too large or not significantly better than simpler methods. Essentially we’d be trying to hand-craft our own mini-embedding model for the domain.
*   **Static Word Embeddings & Averaging:** Instead of a huge transformer, one could use smaller pretrained word vectors (like GloVe or Word2Vec) for known vocabulary words. We then average or sum those to represent the sentence (this is what we attempted that gave 45% accuracy). There’s a known limitation that averaging drops word order and often underestimates the importance of certain words, leading to semantic dilution. However, some improvements could be made: e.g., using **IDF-weighted embedding averages** (a common trick in information retrieval to favor rare words in the average). Also, using a domain-specific mini embedding set could help – for example, training Word2Vec on fantasy text (item and action descriptions) so that it knows *sword*, *blade*, *dagger* cluster together. But training Word2Vec might be difficult with limited in-game text.
*   **Feature Hashing for Text:** The “hashing trick” (Weinberger *et al*., 2009) projects high-dimensional one-hot vectors into a smaller dimensional space by using a hash function. This is used in libraries like scikit-learn to reduce memory usage of text features. We could hash each token (or token bigram) into, say, a 1000-length bit vector (or int counts) and use that as a representation. Two similar phrases will have more overlapping hash bits by chance (especially if using MinHash). However, without learning, this might not be much better than Jaccard; it’s more an engineering solution for handling very large feature spaces, which isn’t our bottleneck here.

**Key Findings:** The overarching theme is that **algorithmic similarity techniques can substitute for certain aspects of embeddings** but often at a cost in accuracy:

*   LSH/MinHash can rapidly identify high Jaccard-similar items in massive collections, but it doesn’t actually improve the similarity measure; it just makes it faster. Some recent works (like Andoni *et al*., 2015) combined LSH with product quantization for efficient approximate *cosine* similarity search, but those still assume you have vector representations.
*   Random projection methods (like random indexing by Sahlgren 2005) showed you can accumulate context in a vector space without heavy learning: assign random vectors to words and sum them for contexts, periodically re-normalizing. This was used in early distributional semantics as a memory-light alternative to SVD/LSA. It maintains some notion of semantic similarity, but not as well as learned embeddings.
*   Knowledge distillation can compress big models into smaller ones for faster inference. For example, **DistilBERT** (Sanh et al., 2019) is a \~66M parameter model that retains \~97% of BERT’s performance with half the size, through distillation. Even smaller specialized student models (tens of MBs) have been trained for tasks, but our constraint is not just size – it’s the *no external dependency* rule. Even a 10MB distilled model can’t run in pure Lua. However, one could “distill” knowledge into structured data or rules. A curious example: Microsoft’s old ENTRA system (1990s) learned decision rules from data – we could imagine using a decision tree classifier to predict the command from features of the input. But with 4,200 classes (verb+noun pairs), a decision tree might be huge and essentially mimic the phrase table we already have. Still, using a classifier to identify the verb (48 classes) could be done with a simple decision tree or even hardcoded rules (if we had a way to get features like “does input contain a synonym of *attack*?” etc.). Such a classifier could be induced from data (if we generate many sample sentences for each verb via an LLM and then train a lightweight model on bag-of-words features). This borders on using ML at runtime, but if the model is simple enough (e.g., a bunch of `if-else` or a small matrix multiplication), it might be permissible.

**Direct Applicability:** Given our constraints, the most practical non-neural embeddings alternatives are those that can be precomputed and used with simple math:

*   **Precomputed Key-Word Vectors:** As an intermediate step, we could precompute a short vector for each command phrase using something like random indexing or a custom 2-layer neural network (executed at build-time). But an easier approach: compute a single vector per *word* (e.g., from a small embedding set or even assigning a random high-dimensional bit vector to each word) and sum them for phrases. Then do the same for input at runtime and compare vectors with cosine. This is essentially what we attempted with static word embeddings; results were subpar (45%). One reason is that some words like “get” appear across many phrases, diluting the vector. Weighting by IDF or using a larger vector dimension might help slightly. However, expecting a dramatic accuracy improvement from such hand-designed embeddings is optimistic – neural models excel precisely because they capture nuances that simple vectors miss.
*   **LSH/MinHash for speed:** If we ever scale to tens of thousands of phrases (say, if the game world grows or we include more NPC dialogue parsing), implementing an LSH for Jaccard similarity could keep the lookup time low (sub-linear). We could generate, for each phrase, a few 64-bit MinHash signatures (each a hash of a random permutation of tokens) at build time. At runtime, hash the input similarly and only compare with phrases sharing a signature. However, for now, our performance is acceptable, so this is low priority – it’s nice to know we have options if needed.
*   **Feature Hashing**: Instead of storing every token as a string, we could hash tokens to integers to save space and compare quickly. But 4.6k×(average \~3 tokens) ≈ 14k tokens total – even storing them as short strings is not a big issue memory-wise. If we did have memory constraints (e.g., on a small device), we could compress the phrase index by replacing strings with int IDs via a hash table or perfect hashing. This would be an engineering improvement without affecting accuracy.

**Estimated Improvement:** The techniques in this section (LSH, random projection, feature hashing) are mostly about *efficiency and feasibility* rather than significant accuracy gains. They ensure we can handle similarity search without a heavy model. The one accuracy booster here is the notion of **embedding distillation** – which is highly experimental. If, hypothetically, we succeeded in training a tiny model to emulate GTE-tiny’s behavior for our domain, we could approach the \~90% semantic matching accuracy that GTE-tiny itself might achieve. That could mean pushing our Tier 2 success close to the upper 80%s (since GTE-tiny was estimated \~90% on our tests). However, this is a non-trivial path and could be risky if the student model mis-predicts. In contrast, simpler methods like soft cosine (discussed above) probably yield smaller but safer gains (\~10% relative improvement).

**Implementation Complexity:** **Effort: high (for advanced methods), low (for simpler ones).** LSH/MinHash and feature hashing are well-known algorithms that are not too hard to code, but they aren’t necessary unless we have performance issues. Random projection or custom mini-embeddings would be a research project in itself: we’d need to experiment with different techniques and evaluate on our tasks. That’s significant work and may not outperform the more direct methods from Section 3. Knowledge distillation to a small model is also a heavy lift requiring ML expertise and would violate “no runtime inference” unless the distilled model can be represented as simple rules or matrix ops implemented in Lua (which is theoretically possible – e.g., one could embed a small matrix multiplication and nonlinear function in Lua – but that starts to blur the line of “no external dependencies”). Given our requirements, the safer route is to stick with symbolic or statistical methods (e.g., similarity matrices, dictionaries, weighting) rather than trying to reinvent neural embedding inside Lua. If extreme optimization is needed, then exploring these could be worthwhile, but otherwise the return on investment is questionable.

## 5. How Coding Assistants Shape Input: Architecture Deep Dive

**Key Insights from Industry:** Modern coding assistants illustrate how to **prepare and refine input context** for an AI, which has parallels to how our parser might pre-process and narrow down player commands. According to GitHub’s documentation and blog posts, Copilot and similar systems use a multi-layer prompt structure: a **system layer** (with instructions and tools), followed by a **context layer** (with relevant code snippets, file structure, etc.), and then the **user query**. While we don’t use an LLM at runtime, we can mimic some of these layers in our system: [\[dev.to\]](https://dev.to/seiwan-maikuma/a-deep-dive-into-github-copilot-agent-modes-prompt-structure-2i4g)

*   **System/Policy Layer:** In Copilot, the system prompt contains rules and identity (e.g. “You are an AI coder…” plus tool guidelines). Our parser has a fixed “policy” implicitly defined by code: e.g., it knows the list of verbs and objects and how to handle ambiguity. We’ve essentially hardcoded the “system prompt” by designing the parser’s tiered logic. Not much changes here, except ensuring the parser follows strict rules (like never guessing an unsupported action – analogously how Copilot’s system prompt makes it refuse disallowed outputs). *   **Context/Workspace Layer:** This is where coding assistants inject relevant code. They maintain **indices** of code and leverage both static analysis and retrieval. For example, Copilot uses a local index of function names, comments, etc., to find what code the question likely refers to. Cursor uses Tree-sitter to parse code into an AST so that each function or class definition can be treated as a chunk, ensuring meaningful retrieval units. For our parser, the equivalent is the game’s **current state and recent history**. We already have Tier 4 using a “recency stack” to resolve pronouns (so if the player types “grab that”, we look up the last mentioned object). We might enhance context use by also considering the current room’s available objects and NPCs as implicit context. If a command mentions **no noun** or an ambiguous noun, we could infer it by context (similar to how a coding assistant might guess what “run it” refers to by looking at the nearest code mention). For instance, if the player just examined a *book* and then says “read it”, we know “it” = book. Or if the player is in a room with only one readable item, “read” could default to that item. This is more a *game design* decision (some parsers already do this kind of inference), but it’s inspired by how AI assistants use conversation history and environment state as part of the prompt. *   **Heuristic Ranking:** Coding assistants also apply special heuristics, e.g., *prefer code from the same file or function the user is in, even if a global search might find other matches*. Similarly, our parser can have custom heuristics: e.g., if two candidate commands score similarly by similarity, prefer the one involving an object that is actually present and reachable to the player (which we likely do in later validation, but we could integrate availability into the score as a soft boost). Another heuristic: if the input contains extra words that match none of the known verbs/nouns, perhaps the player is chatting or expressing an unrecognized action – we might then route to a default response (“I don’t understand that command”) faster or consider if it’s an out-of-world command (like meta-commands “save game”, “help”, etc.). In essence, context can tell us when to apply certain parsing strategies.
*   **Use of Tools in Prompt:** In AI agents (like Copilot’s hidden system tools), the assistant is instructed to use certain tools to retrieve information when needed (e.g., search the web, read file, etc.). In our static environment, **we are the tool** – our Tier 2 is the “Semantic Matcher” tool. We might conceptualize our pipeline as a series of tools: exact match tool, semantic match tool, idiom fixer, pronoun resolver, etc., applied in sequence (which is exactly what our 5-tier system is). The key is that each layer adds context or transforms the input to help the next layer. [\[dev.to\]](https://dev.to/seiwan-maikuma/a-deep-dive-into-github-copilot-agent-modes-prompt-structure-2i4g)

**Key Findings:** The architectural deep-dives reveal that **successful AI assistants carefully balance local processing and remote (model) calls**:

*   Copilot’s local VS Code extension is responsible for gathering context before any API call to the LLM – analogous to our offline precomputation of phrases and at-runtime selection. By doing as much work as possible locally (or in our case, at build time and via lightweight code), they reduce dependence on the heavy model. This reinforces that our approach of build-time computation (generating phrases, computing weights/similarities) is on the right track. The takeaway is our pipeline should continue to handle easy cases with simple, high-precision methods (exact alias lookups, known idioms) and reserve the more “creative” interpretation for when those fail. This avoids false positives and keeps interactions predictable.

**Direct Applicability:** The architectural ideas from coding assistants support many points made in previous sections:

*   Maintain a “workspace-like” context: in a text adventure, the **game state is the context**. We should make use of it. For example, if the user mentions “cook dinner” and the player is in a kitchen with a stove and ingredients, the parser might infer it should map to the “cook *food*” action. In contrast, if the player is in a forest, “cook dinner” might default to an error or a different action (maybe setting up a campfire). This is analogous to how Copilot treats `@workspace` – it scopes the query to current context. Implementing this might involve tagging each phrase variant with the context (location or state) in which it’s valid, and boosting/deprioritizing based on the current game context at runtime. This goes beyond pure NLP matching into game logic – but it can dramatically reduce ambiguity. *   Use a **priority ranking** of potential matches: If our similarity scoring yields multiple candidates above a threshold, apply domain-specific tiebreakers. E.g., prefer an exact match on verb if one exists; or if one candidate refers to a visible object and another to an absent object, choose the former. *   **Minimal necessary context:** just as prompt packing avoids irrelevant code, our parser shouldn’t consider irrelevant phrases. For example, if the input contains the token “candle”, there’s no need to consider phrases that don’t have “candle” or a known synonym in them. This is like the **metadata filtering** that Cursor’s pipeline does (using file path metadata to filter by directory or file when the query contains a filename). We can treat each noun as a “metadata tag” and only compare against phrases containing that noun (or its synonym set). **Estimated Improvement:** Leveraging context (game state and recent references) can reduce misinterpretations and increase success in ambiguous commands by perhaps another few percentage points. It mostly affects *precision* (making sure we pick the right target when multiple are possible) rather than recall. The main gains in recall/coverage come from the earlier sections (synonym expansion, soft match). More importantly, it improves **player experience** by making the parser feel smarter and more aligned with what the player *intends* given the in-game situation.

**Implementation Complexity:** **Effort: low-to-moderate.** Many of these are logic tweaks:

*   Tagging phrase variants with context (if not already done) might require adding metadata to phrases (like “valid in location X” or “requires object Y to be present”). The game likely already has some of this logic in the action handling, but doing it at parse time is trickier. We would need access to the game state in the parsing function to filter or rank matches. If the parser is separate from game logic, this could complicate design.
*   Building a stop-word list for the game and implementing removal or down-weighting is trivial.
*   Maintaining recent references (like we do for pronouns) can be extended to a more general mechanism for implicit object references. This might involve storing the last mentioned object for certain verbs. For example, after “open door”, if next command is just “close it”, we know “it” is the door. This is partly done, but extending it (like “attack again” meaning attack the same target as before) could be added.
*   None of this involves heavy computation, just careful design and testing to ensure the parser doesn’t make incorrect assumptions. We have to ensure these heuristics don’t misfire (for instance, assuming context that isn’t intended by the player).
*   Overall, these changes are incremental and can be implemented and tested individually.

## 6. Practical Hybrid Approaches for Our Constraints

Bringing it all together, we identify a few **concrete techniques** that emerge from the above research, which are feasible under our no-runtime-neural constraint and likely to yield significant improvements. Below, we list these recommended approaches, along with their expected impact and implementation effort, then provide a prioritized ranking.

### Effort-to-Impact Priority

1. **Synonym & Slang Expansion (Offline)** — Use WordNet, thesauri, or LLMs at build-time to expand each verb and noun with common synonyms, slang, and paraphrase templates. This directly addresses the "novel phrasing" gap. Impact: High improvement in recall (5–15% overall). Effort: Moderate (one-time corpus curation).

2. **BM25 Lexical Scoring** — Replace or augment Jaccard with a TF-IDF/BM25 based similarity for matching commands. Better handling of extra words and differing lengths. Impact: Moderate (~5% gain in accuracy; more robust matching). Effort: Low (simple formula in Lua).

3. **Soft Matching via Word Similarity** — Implement soft cosine/Jaccard using a precomputed word similarity matrix from embeddings. This allows partial credit for similar words ("grab" ≈ "get"). Impact: High on previously missed synonyms (~10% gain, particularly in paraphrase cases). Effort: Moderate (compute matrix offline; small runtime cost).

4. **Context-Aware Filtering** — Leverage game context and recent references to filter and rank candidate commands (e.g., focus on current room's items, last mentioned object) similar to how Copilot uses the @workspace context. Impact: Low-to-moderate (improves precision, resolves ambiguities). Effort: Low (use existing game state info).

5. **Inverted Index & Fuzzy Search** — Build an inverted index of tokens to candidate phrases to limit search scope (only consider phrases sharing a rare token with input). Integrate a secondary fuzzy match for when no good match is found. Impact: Low (mainly performance & backup for edge cases). Effort: Low.

### Effort-to-Impact Priority

Considering the above, we recommend proceeding in this order of priority:

1.  **Synonym/Paraphrase Expansion (Offline):** *High impact, moderate effort.* This directly targets the largest source of error (unrecognized phrasing). It’s supported by research demonstrating improved recall in similar settings. The implementation is offline, so it won’t affect runtime performance. We should start by integrating WordNet or a curated synonym list to cover obvious gaps (“gimme”→“get”, “attack”→“hit”, etc.), yielding immediate gains. [\[risame.github.io\]](https://risame.github.io/sun/query.pdf)

2.  **Soft Similarity Matching:** *High impact, moderate effort.* By utilizing a word similarity matrix from GTE-tiny or another source, we capture subtle relationships between words without running a model live. This will help with cases where players use related terms (e.g., “use sword” vs our “attack troll with sword” command). It requires some math but no new dependencies at runtime. Testing on our benchmark can fine-tune the contribution of soft matches to avoid false positives.

3.  **BM25/Inverted Index Implementation:** *Moderate impact, low effort.* Replacing Jaccard with a more principled lexical similarity (and using an inverted index for efficiency) will refine matching, especially for longer inputs. It’s straightforward to implement and can coexist with the above methods. Given its ease, it’s worth doing, though the gains might be smaller than synonyms/soft matching. Even a few percent improvement and better robustness to junk words is worthwhile.

4.  **Contextual Ranking Heuristics:** *Moderate impact, low effort.* These enhancements (using location, recent mentions) align the parser with player intent in ambiguous situations. They won’t fix completely new phrasings, but they will reduce errors like applying the right verb to the wrong object. Since the effort is small (tapping into existing game state), it’s a good polish step after tackling the bigger issues above.

5.  **Advanced Techniques (Learned mini-embeddings or LSH):** *Low impact, higher effort.* These are not top priorities because our scale is small and simpler solutions suffice. We should only consider them if we either: (a) dramatically expand the game’s vocabulary and need better performance, or (b) exhaust simpler methods and still need accuracy boosts. For instance, if one day the game includes a much larger command set or even natural-language dialogues, then investing in a compact semantic parser model (perhaps a bespoke classification model distilled from GPT on a large corpus of fantasy text) might make sense. Until then, the complexity of these approaches isn’t justified by the incremental benefit.

## Conclusion

By learning from AI coding assistants and IR research, we can substantially enhance our text adventure parser’s ability to understand players’ intentions **without violating the no-runtime-ML constraint**. The primary theme is **smarter retrieval and matching**: use better lexical scoring (BM25) and precomputed semantic knowledge (synonym lists, embedding-driven similarities) to bridge the gap between a player’s freeform language and the game’s understood commands. These techniques are grounded in decades of research: from classic IR methods like TF-IDF and Rocchio’s query expansion to modern advances like soft cosine similarity and hybrid retrieval strategies in RAG systems. By implementing the recommended changes – **offline synonym expansion, soft matching, improved scoring, and context-aware heuristics** – we estimate the parser’s overall command resolution accuracy could improve from \~68% into the mid-70s or higher, with “hard” phrasings significantly better handled. These enhancements will make the game more forgiving and intuitive, providing a smoother experience for players who naturally phrase commands in unexpected ways. All this can be achieved with manageable development effort and no new runtime dependencies, which is a testament to how traditional IR and clever prompt engineering can complement or even substitute for heavy AI models in resource-constrained environments. By adopting a *retrieval-augmented parsing* approach, our text adventure can better understand the many ways a player might say the same thing – just as Copilot intelligently interprets a programmer’s request by leveraging all available context.
