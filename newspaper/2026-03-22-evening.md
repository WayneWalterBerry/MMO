## 🎮 SECTION: PHASE 3 — FIVE MAJOR FEATURES IN ONE SESSION

### The Smithers Blitz (21:44Z)

Smithers sat down at 21:44Z with five design documents from CBG and stood up with five implemented, tested, and deployed features. This is the session that will be talked about.

### Feature 1: Hit Verb — Violence Comes to the MMO

**Verbs:** `hit`, `punch`, `bash`, `bonk`, `thump`

The hit system introduces the game's first combat mechanic — and it starts with self-infliction (V1). Players can hit themselves. This is not a bug. This is *game design*.

**Body Area Targeting:**

```lua
-- Body areas with severity effects
local body_areas = {
    head  = { ko_chance = true, duration = {10, 15} },  -- knockout 10-15 turns
    limb  = { bruise = true, duration = {5, 10} },       -- bruise 5-10 turns
    torso = { winded = true, duration = {3, 5} },        -- winded 3-5 turns
}

-- Armor reduces severity
if player:wearing_armor_on(area) then
    severity = severity * armor.reduction_factor
end
```

**Verb Disambiguation:**
- `strike` + body area → **hit** (combat)
- `strike` + flint → **fire-making** (survival)
- `smash` remains aliased to `break` (object destruction, not combat)

**Decisions Filed:** D-HIT001, D-HIT002, D-HIT003

### Feature 2: Unconsciousness State Machine — Lights Out

The game now models unconsciousness as a full state machine:

```
conscious → (trauma) → unconscious → (recovery) → conscious
                              ↓
                          (severity)
                              ↓
                     death (if injuries worsen)
```

**Severity-Based Duration:**
- Minor blow: 3-5 turns unconscious
- Moderate blow: 5-15 turns unconscious
- Severe blow: 15-25 turns unconscious

**The Consciousness Gate:**

This is the architectural breakthrough. The game loop now checks consciousness *before* reading player input:

```lua
-- Game loop with consciousness gate
while game.running do
    if player.state == "unconscious" then
        -- Skip input reading entirely
        player:tick_unconscious()  -- injuries tick, recovery progresses
        if player:check_death() then
            game:end("You never regain consciousness.")
        end
    else
        local input = read_player_input()
        process_command(input)
    end
end
```

**Key Design Insight:** The player doesn't get a "you are unconscious" prompt. They simply *don't get prompted*. The game ticks time forward, injuries accumulate, and eventually they either wake up or die. The wake-up narration tells them what happened while they were out.

**Decision Filed:** D-CONSC-GATE

### Feature 3: Sleep Injury Fix — No More Safe Havens

**The Bug:** The `sleep` verb was a get-out-of-jail-free card. While sleeping, the injury system's `injury_mod.tick()` was never called. A player bleeding to death could type `sleep` and survive indefinitely.

**The Fix:**

```lua
-- BEFORE: sleep was a safe haven
function verb_sleep(player)
    for i = 1, sleep_duration do
        advance_time(1)
        -- injury_mod.tick() never called!
    end
    player:wake_up()
end

-- AFTER: sleep ticks injuries
function verb_sleep(player)
    for i = 1, sleep_duration do
        advance_time(1)
        injury_mod.tick(player)  -- bleeding continues while sleeping
        if player:check_death() then
            game:end("You never wake up.")
            return
        end
    end
    player:wake_up()
end
```

**"You never wake up."** — Four words. Maximum impact. Death during sleep is now possible and narratively powerful.

**Decision Filed:** D-SLEEP-INJURY

### Feature 4: Appearance Subsystem — How Do You Look?

**File:** `src/engine/player/appearance.lua`

A pure stateless function that generates player appearance descriptions. No stored state — it reads the player's current equipment, injuries, and health, then composes a description on every call.

**Layer-Based Rendering:**

```lua
-- Appearance layers (rendered top to bottom)
local layers = {
    "head",     -- helmet, hat, head injuries
    "torso",    -- armor, shirt, torso wounds
    "arms",     -- bracers, arm injuries
    "hands",    -- gloves, ring, hand state
    "legs",     -- greaves, pants, leg injuries
    "feet",     -- boots, bare feet
    "overall",  -- general health tier
}
```

**Health Tiers:**
- 🟢 **Fresh** — "You look healthy and ready for adventure"
- 🟡 **Minor** — "You have a few scrapes and bruises"
- 🟠 **Moderate** — "You're battered and showing signs of wear"
- 🔴 **Critical** — "You look terrible — blood-soaked and barely standing"

**Injury Phrase Composer:** Combines active injuries into natural-language descriptions:
- One bruised arm: "Your left arm is bruised"
- Multiple injuries: "Your left arm is bruised, and blood seeps from a wound on your torso"

**Multiplayer-Ready API:** The function takes a player object as input and returns a description string. Any player can describe any other player — designed for future multiplayer from day one.

**Decision Filed:** D-APP-STATELESS

### Feature 5: Mirror Integration — Vanity Meets Gameplay

Objects with `is_mirror = true` now route `examine` to the appearance subsystem:

```lua
-- Mirror integration
function on_examine(player, object)
    if object.is_mirror then
        local desc = appearance.describe(player)
        emit("You gaze into the " .. object.name .. ".")
        emit(desc)
        -- Vanity context narration
        if player.health_tier == "fresh" then
            emit("Not bad, if you do say so yourself.")
        elseif player.health_tier == "critical" then
            emit("The face staring back looks like it's seen better days.")
        end
    else
        emit(object.description)
    end
end
```

**Impact:** Mirrors are now functional gameplay objects. Players can check their status, see equipment, and assess injuries — all through the `examine mirror` command.

## 🏗️ SECTION: SPATIAL RELATIONSHIPS — DESIGNED AND ARCHITECTED IN ONE EVENING

### The Problem

The engine had ONE spatial relationship: `contains`. A table contains objects. A chest contains objects. A room contains objects. But players think in richer terms:

- "Look **on** the table" (surface — visible)
- "Look **under** the table" (beneath — possibly hidden)
- "Look **behind** the tapestry" (concealed — definitely hidden)
- "Look **inside** the chest" (container — may be locked)

One relationship cannot model four spatial concepts.

### CBG's Design: Hiding vs. On-Top-Of

CBG (Design Lead) designed the spatial distinction:

```
VISIBLE RELATIONSHIPS:
  on_top_of  → Objects are visible in room description
  beside     → Objects are visible, contextually adjacent

HIDDEN RELATIONSHIPS:
  under      → Objects NOT visible until player looks under
  behind     → Objects NOT visible until player looks behind
  inside     → Objects NOT visible until container is opened
```

**Decision Filed:** D-SPATIAL-HIDE

**Key Insight:** Visibility is the defining attribute. "On the table" means the player sees it. "Under the table" means the player must actively search.

### Bart's Architecture: Object-Level Metadata

Bart translated CBG's design into engine architecture:

```lua
-- Object spatial metadata
object.spatial = {
    relation = "under",       -- spatial relationship to parent
    parent = "table_001",     -- what this object is relative to
    visible = false,          -- derived from relation type
    discoverable = true,      -- can be found by searching
}

-- Traverse.lua bug discovered during architecture work
-- traverse was treating all children as visible — fixed
```

**Decision Filed:** D-SPATIAL-ARCH

**Discovery:** While architecting spatial metadata, Bart found a bug in `traverse.lua` — the room traversal code was listing ALL child objects regardless of visibility. This would have made every hidden object appear in room descriptions. Caught before it shipped.
