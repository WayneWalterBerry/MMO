## 📰 OP-ED: "FUZZY MATCHING MUST BE A FIRST-CLASS CITIZEN IN EVERY VERB"
### *By Smithers, Parser Lead*

For months, we've treated fuzzy noun resolution as a special feature for LOOK and GET. A fallback. Something we bolted on when the exact match failed.

Tonight's playtest showed us why this is backwards.

Issue #181 wasn't "drop" failing to find a sack. It was "drop spittoon" — a partial name that should have **obviously** matched the only sack-like object in the room. But it didn't, because the drop handler wasn't calling the fuzzy resolver. The player got "object not found" instead of a graceful fallback.

Then Issue #182: Three kinds of brokenness at once:
- Color descriptors ("drop red sack") were ignored
- Synonyms were unrecognized ("dump" → "drop")
- When two objects matched, no disambiguation prompt appeared

These aren't parser bugs. These are **verb handler bugs**. Every verb had its own resolution logic (or lacked it entirely).

### The Problem: Inconsistent Entry Points

Right now, fuzzy matching lives in tier 5 of the parser pipeline — a fallback when tiers 1–4 all fail. That makes sense at the **global level**. But inside individual verb handlers, we're doing this:

```lua
verbs.look = function(context, noun)
  local obj = context.registry:find_by_keyword(noun)
  if not obj then
    obj = fuzzy_fallback(noun)  -- explicitly call fuzzy
  end
  print(obj.description)
end

verbs.drop = function(context, noun)
  local obj = context.registry:find_by_keyword(noun)
  if not obj then
    print("You don't have that.")
    return  -- NO FALLBACK
  end
  context.inventory:remove(obj)
end
```

Drop doesn't even try to recover. The player's mental model is "I can name things any way I want" (because LOOK works that way). But DROP betrays that model.

### The Fix: Verb Handlers Must Resolve Consistently

We need a unified resolution path that every verb uses:

```lua
local function resolve_object(context, noun, scope)
  -- Tier 1: Exact keyword match
  local obj = context.registry:find_by_keyword(noun, scope)
  if obj then return obj end
  
  -- Tier 2: Fuzzy match (adjectives, partial names, synonyms)
  obj = fuzzy_resolve(noun, scope)
  if obj then return obj end
  
  -- Tier 3: Disambiguation prompt (multiple matches)
  local candidates = fuzzy_candidates(noun, scope)
  if #candidates > 1 then
    return prompt_user_to_disambiguate(candidates)
  end
  
  return nil  -- Not found
end

verbs.drop = function(context, noun)
  local obj = resolve_object(context, noun, context.current_inventory)
  if not obj then
    print("You don't have that.")
    return
  end
  context.inventory:remove(obj)
end
```

Every verb handler gets the same three-tier resolution. Every verb gets access to adjective scoring, partial matching, synonym resolution, and disambiguation. No special cases.

### Why This Matters

Players don't think in terms of "exact keyword match" vs. "fuzzy match." They think: **I name objects in the world, and the game understands me.** If LOOK understands me but DROP doesn't, the game feels broken.

More fundamentally: Fuzzy matching isn't a parser feature. It's a **verb feature**. The parser's job is syntax. The verb handler's job is semantics — understanding what the player *meant*, not just what they *typed*.

By making fuzzy resolution a first-class citizen in every verb handler, we're saying: **Understanding the player is part of the verb's contract.** Not optional. Not a fallback. Baked in.

Tonight we fixed five verb handlers. But the architectural pattern goes deeper. Every handler we write going forward should start with this. Every verb in the system should resolve objects the same way.

That's not extra work. That's consistency.

### The Payoff

Issue #181 and #182 were 13 + 24 = 37 test cases. Those test cases now catch:
- Missing fuzzy fallbacks
- Broken adjective scoring
- Synonym gaps
- Disambiguation failures

Any new verb that violates the pattern breaks the tests. We've codified the expectation into the test suite.

In a month, when players playtest this game, they won't hit a "you don't have that" error when they say "drop the red sack." They'll get what they expect: the game understanding them.

That's when the game stops feeling like a puzzle and starts feeling like a conversation.
