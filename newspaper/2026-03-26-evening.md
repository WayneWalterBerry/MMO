## 🍖 FOOD SYSTEM DESIGN: Freestyle Breakthrough

### The Design Session (4 Systems in 15 Minutes)

**Frink** (Research) delivered 185KB of comparative analysis across 15+ adventure games, establishing design space. **Comic Book Guy** wrote the 25KB `food-system-plan.md` (expanded to 29KB with directives).

Then Wayne Berry walked to the whiteboard and, without notes, designed **four interconnected systems**:

#### 1. **Creature Death → Food via D-14 Mutation**
When a rat dies, `rat.lua` hot-swaps to `dead-rat.lua`. The creature becomes an object: no state flags, no conditional rendering — the code IS the death. Carrion can be eaten, buried, or left to decay (time-based transformation).

#### 2. **Cooking-as-Craft (Verb + Tool → Mutation)**
```
cook <dead-rat> with <fire_source>
  → dead-rat.lua mutates to cooked-rat.lua
  → gains nutritional properties
  → loses disease transmission risk
```
Reuses the verb pipeline and fire_source capability system. No new engine code required.

#### 3. **Creature Inventory + Loot Drops**
Rats carry food in their own containment trees. When killed, inventory drops intact (reusing containment hierarchy). A rat carrying poison mushrooms now represents ecological complexity.

#### 4. **Eat Causes Injuries (Effect Pipeline)**
```
eat <raw-rat>
  → triggers "food_consumed" effect
  → resolves injury list: [rabies, trichinosis, salmonella]
  → applies based on creature_hygiene and player_health
```
Integrates seamlessly with existing injury subsystem.

### Architecture Decision
**Food = metadata trait, not template**. Objects can declare `is_edible`, `disease_vector`, `nutrition_value` without requiring multi-inheritance. Contained within existing template system.

| Cooking | Verb + capability | verbs.lua, effects.lua | ~20 lines |
| Inventory | Containment tree | containment.lua | 0 lines |
| Disease injury | Effects pipeline | effects.lua, injuries.lua | 0 lines |

## 📰 OP-ED: "Constraint-Driven Design: Why Wayne's Freestyle Worked"
**By Smithers, Parser Architect**

I've watched the team tackle the food system three times: once as a template (bloated), once as an NPC behavior subsystem (scope creep), and once as a table-driven data layer (incoherent).

Then Wayne walked up and said, "What if food isn't a *thing* — what if it's a *property that things have*?"

The insight was deceptively simple: **constraints force clarity**. We couldn't add templates (no multi-inheritance). We couldn't add new engine systems (code freeze). We couldn't add state flags (D-14 forbids them). So we asked: *How do we solve food within what we already have?*

The answer was waiting in the architecture all along. Mutations for death. Effects for injuries. Containment for inventory. Verbs for actions.

Food didn't need *design*; it needed *alignment*. Wayne found the alignment. That's the difference between a designer and an architect: designers imagine new worlds; architects see the hidden patterns in the world they have.

The 15-minute freestyle worked because the foundation was solid. Constraint-driven design rewards preparation — the team had already built the tools. Wayne just showed us how they fit together.
