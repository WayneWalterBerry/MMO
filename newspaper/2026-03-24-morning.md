## ⚙️ SECTION: ARMOR SYSTEM ARCHITECTURE — MATERIAL-DERIVED PROTECTION

### Bart: Phase A Complete — Material Properties Drive Armor

Yesterday morning, Bart tackled the armor system design question that had been deferred since Phase 4: *Should armor protection be hardcoded per-object, or derived from material properties?*

The answer: **material properties all the way down**.

Armor protection now derives from three material factors:
1. **Material Hardness** (resistance to deformation)
2. **Material Density** (mass per volume, slows kinetic energy)
3. **Thickness** (depth of material, absorbs impact)

**The Formula:**

```lua
protection = (hardness * 0.4) + (density * 0.3) + (thickness * 0.3)
-- Result range: 1–10 (soft leather) to (steel plate)
```

**Example Calculations:**

| Material | Hardness | Density | Thickness | Protection |
|----------|----------|---------|-----------|------------|
| Tallow Candle | 2 | 1 | 1 | **1.3** (soft, no armor value) |
| Leather Cloak | 3 | 2 | 2 | **2.2** (basic protection) |
| Wool Tunic | 2 | 3 | 3 | **2.7** (insulating, minor armor) |
| Ceramic Pot | 6 | 4 | 4 | **4.8** (brittle, moderate) |
| Iron Breastplate | 8 | 8 | 6 | **7.2** (heavy, professional) |
| Steel Sword (worn as helmet) | 9 | 8 | 5 | **7.4** (overkill, but it works) |

**Why This Matters (Principle 9):**

This enforces Core Principle 9 (Material Consistency). A wooden shield behaves like wood — good against slashing, weak against crushing. A ceramic pot behaves like ceramic — brittle, but deceptively sturdy. A steel breastplate behaves like steel — your classic medieval armor. No special cases. No "this object is magic." Objects inherit their behavior from what they're *made of*.

### Fit Multipliers: Quality of Craftsmanship

Base protection values apply to a perfect fit. Real armor varies:

```lua
worn_protection = base_protection * fit_multiplier

fit_multipliers = {
    makeshift = 0.5,   -- (you wrapped a cloth around yourself)
    fitted = 1.0,      -- (proper armor, proper fit)
    masterwork = 1.2,  -- (master craftsman, perfect balance)
}
```

**Example:**
- Leather cloak (fitted): 2.2 × 1.0 = **2.2 protection**
- Leather cloak (makeshift): 2.2 × 0.5 = **1.1 protection**
- Leather cloak (masterwork): 2.2 × 1.2 = **2.6 protection**

### State Multipliers: Degradation Over Time

Armor degrades when damaged. The state multiplier reflects this:

```lua
effective_protection = worn_protection * state_multiplier

state_multipliers = {
    intact = 1.0,      -- (new condition)
    cracked = 0.7,     -- (damaged but functional)
    shattered = 0.0,   -- (destroyed, no protection)
}
```

**Example:**
- Iron breastplate (intact): 7.2 × 1.0 = **7.2 protection**
- Iron breastplate (cracked): 7.2 × 0.7 = **5.0 protection**
- Iron breastplate (shattered): 7.2 × 0.0 = **0 protection**

### Equipment Event Hooks — on_wear, on_remove_worn

Objects can now react to being worn/removed:

```lua
return {
    id = "iron-breastplate",
    template = "armor",
    -- ... other properties ...
    
    on_wear = function(context, wearer)
        -- Fires when player wears this armor
        context:narrate("The armor settles across your shoulders, heavy and reassuring.")
        context.player:add_protection(7.2)
    end,
    
    on_remove_worn = function(context, wearer)
        -- Fires when player removes this armor
        context:narrate("The weight lifts. You feel suddenly lighter.")
        context.player:remove_protection(7.2)
    end,
}
```

**Implementation:** Smithers added hook dispatch in the `wear` and `remove` verbs. Objects declare `on_wear` and `on_remove_worn` callbacks; the engine executes them when the state changes.

### Instance-Level Flavor Text: Event_Output System

Objects can now have per-instance one-shot flavor text that fires when something happens, then self-removes:

```lua
-- In a room definition, per-instance:
{ 
    id = "iron-breastplate",
    type_id = "iron-breastplate-guid",
    event_output = {
        on_examine = "The breastplate gleams with an almost sacred quality, etched with your initials."
    }
}
```

**How It Works:**

1. First time player examines this instance: flavor text fires, narrated.
2. `event_output` is marked as consumed in the instance state.
3. Future examinations: object's standard description used.

**Design Rationale:** Instances are unique (each player finds a *different* breastplate). Some instances deserve unique flavor text. Event_output provides that without requiring mutation (code rewrite) for simple flavor variations.

## 🔧 SECTION: SMITHERS' P1 PARSER BUG CLUSTER — 7 ISSUES, ONE COMMIT

Smithers demolished the hit verb synonym cluster and related parser issues in a single surgical commit:

### The Bug Cluster (#137–145, #156)

| Bug | Problem | Category |
|-----|---------|----------|
| **#137** | "smack" redirects to hit verb but doesn't find noun | Synonym dispatch |
| **#138** | "bang" parsed as verb but noun resolves to wrong object | Synonym dispatch |
| **#139** | "slap" synonym works for player, fails for objects | Synonym dispatch |
| **#140** | "whack" recognized but case-sensitivity breaks dispatch | Synonym dispatch |
| **#141** | "headbutt" parsed, but "head" target resolves to container | Noun resolution |
| **#142** | "bonk" on head resolves to helm instead of head part | Keyword collision |
| **#143** | "toss" doesn't redirect to "drop" (missing synonym) | Synonym dispatch |
| **#145** | Prepositional suffixes ("at", "on", "with") corrupt parsed target | Parsing |
| **#146** | "throw" synonym works 50% of the time (context-dependent) | Synonym dispatch |
| **#154** | Decorative suffix stripping ("the" in "smack the head") breaks noun matching | Parsing |
| **#156** | "get X from drawer" fails when X is nested inside drawer | Nested access |

**Root Causes:**

1. **Hit verb synonyms not centralized:** Synonyms (smack, bang, slap, whack, headbutt, bonk) were scattered across multiple verb handlers instead of unified in hit verb dispatch.
2. **Case sensitivity:** Noun matching was case-sensitive; "Head" ≠ "head".
3. **Keyword collisions:** "bonk" → could be "bonk" verb OR "bonk" as part of another noun; resolver didn't disambiguate.
4. **Decorator stripping too aggressive:** Removed articles and prepositions before noun matching, corrupting phrase structure.
5. **Nested container access:** Search system wasn't aware of containers inside containers.

### Smithers' Fix

**Commit:** One comprehensive overhaul of verb dispatch (parser/init.lua).

**Before:**

```lua
-- BEFORE: hit verb synonyms scattered
verbs.smack = function(context, noun) 
    -- local implementation
end
verbs.bang = function(context, noun)
    -- different implementation
end
verbs.slap = function(context, noun)
    -- different implementation
end
```

**After:**

```lua
-- AFTER: centralized hit verb with synonym routing
local hit_synonyms = {
    "hit", "strike", "punch",
    "smack", "bang", "slap", "whack",
    "headbutt", "bonk",
    "bash", "bludgeon", "clobber"
}

verbs.hit = function(context, noun)
    local target = resolve_noun(noun, context)
    if not target then
        context:narrate("What do you want to hit?")
        return
    end
    -- unified hit logic
end

-- Register all synonyms to the hit verb
for _, synonym in ipairs(hit_synonyms) do
    verbs[synonym] = verbs.hit
end
```

**Case Normalization:**

```lua
local function resolve_noun(raw_noun, context)
    -- Normalize case
    raw_noun = raw_noun:lower()
    
    -- Remove articles (but preserve structure)
    raw_noun = raw_noun:gsub("^the%s+", "")
    
    -- Remove decorative suffixes (but check context first)
    raw_noun = raw_noun:gsub("%s+(at|on|with|in)$", "")
    
    -- Search with normalized noun
    return context.registry:find_by_keyword(raw_noun)
end
```

**Nested Container Access (#156):**

```lua
-- AFTER: "get X from drawer" now works
local function search_in_container(container_target, item_noun)
    -- Search container's items
    for item in container_target:items() do
        if item:matches_keyword(item_noun) then
            return item
        end
    end
    
    -- If container is nested, also check container's parent
    if container_target.nested then
        return search_in_container(container_target.nested, item_noun)
    end
    
    return nil
end
```

**Result:** All 7 issues closed, plus #154 and #156.

## 🏗️ SECTION: ARCHITECTURE DECISIONS — EQUIPMENT & INSTANCE FLAVOR TEXT

Two architecture decisions formalized this morning:

### D-EQUIP-HOOKS: Equipment Event Callbacks

**Decision:** Objects can declare `on_wear` and `on_remove_worn` callbacks that fire when equipped/removed.

**Rationale:** 
- Objects need to react to being worn (narrative flavor, stat changes, side effects).
- Callbacks are cleaner than mutation-based approaches.
- Allows reusable equipment to have stateful reactions (e.g., armor that announces itself).

**Implementation:**
- Added `on_wear(context, wearer)` and `on_remove_worn(context, wearer)` callback hooks.
- Smithers added dispatch in verbs/init.lua (wear, remove handlers).
- Callbacks fire AFTER state machine transitions, so object is already marked as worn.

**Reference:** docs/design/equipment-system.md

### D-EVENT-OUTPUT: Instance-Level One-Shot Flavor Text

**Decision:** Objects can have per-instance flavor text (event_output) that fires once then self-removes.

**Rationale:**
- Instances are unique; sometimes unique flavor matters (e.g., "This sword belonged to a legendary warrior").
- Event_output allows per-instance customization without mutation.
- Simplifies instance initialization for puzzle flavor.

**Implementation:**
- Added `event_output` table to room instance definitions.
- Event_output contains verb→text mappings (e.g., `on_examine = "Text here"`).
- First verb dispatch: text is narrated, then marked consumed.
- Subsequent dispatches: standard object description used.

**Example:**

```lua
-- In bedroom room definition, instances:
{ id = "iron-breastplate", type_id = "...", 
  event_output = {
    on_examine = "Your family crest is etched into the breastplate.",
    on_wear = "As you don the armor, you remember your oath."
  }
}
```

**Reference:** docs/architecture/objects/instance-overrides.md

## 📰 OP-ED: "MATERIAL PROPERTIES ARE NOT GAMEPLAY TUNING — THEY'RE SIMULATION FIDELITY"
### *By Bart, Engine Architect*

For years, armor systems have been built on a fundamental lie: **hardcoded protection values**. A leather tunic grants +5 defense. A steel breastplate grants +20. These numbers live in balance spreadsheets, get tweaked every patch, and are completely divorced from *what the armor is actually made of*.

This is not game design. This is fantasy role-playing masquerading as simulation.

Today, we shipped something different.

Our armor system doesn't ask "What protection value should leather grant?" It asks a simpler question: "What is leather's tensile strength, mass, and energy-absorption coefficient?" The engine then *derives* protection values from material properties.

A leather tunic grants protection proportional to its thickness, mass, and the energy-dissipation curve of tanned hide. A steel breastplate grants protection based on steel's hardness and the mechanics of force distribution over its curved surface. A *chamber pot* worn as a helmet grants protection based on ceramic's brittleness and impact characteristics — which is to say: *some*, because ceramic breaks easily, but it will absorb a hit.

### Why This Matters

**First:** Consistency. When I design a new object tomorrow, I don't set `armor_value = 15`. I set `material_id = "oak"` and `thickness = 0.03` (meters). The armor system *computes* the protection value from those properties. New objects automatically have sensible values. No spreadsheet tweaks. No balance meetings. The system is self-consistent.

**Second:** Emergent gameplay. The player found a chamber pot and thought, "Could I wear this?" The traditional system would say "no" because chamber pots aren't in the armor table. Our system says "yes" and *calculates what chamber pot armor actually does* based on ceramic properties. The player invented a valid strategy. The system honored it.

**Third:** Scalability. In six months, we'll have 200+ objects. With hardcoded values, every new object is a balance decision. With derived properties, every new object is a material choice, and the system handles the rest.

### The Trade-off (There Always Is One)

Yes, this requires confidence that material properties are predictable. Yes, it means developers must understand material science, not just tweaking spreadsheets. Yes, our chamber pot helmet is legitimately protective, which is ridiculous.

That ridiculousness is the feature. It means the system is *consistent with its own rules*. It doesn't say "leather grants +5" and then break that rule for special cases. It derives protection from properties, always. If that produces weird results, we don't patch the results — we fix the properties.

### The Principle

**Material consistency is more important than balance diversity.** A world where all armor calculations are predictable from material science is more interesting than a world with a thousand balance-tweaked values and hidden special cases.

Core Principle 9: Objects obey real-world material properties. We shipped that today. It means the next developer who adds chainmail doesn't ask me "what armor value?" They ask "what is steel mesh?" The engine does the rest.

That's not a limitation. That's architecture.
