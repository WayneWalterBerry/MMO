## 🏗️ THE ARCHITECTURE (IN ENGLISH)

### The Multiverse Model
- **One universe per player** by default (NOT a shared server world)
- **Opt-in merging** — Players can join together for raids, trading, or events
- **Per-universe fork/merge** — Handled via event sourcing (immutable logs)
- **Copy-on-write optimization** — Storing 1000 players × 10MB universe state ≈ 110MB total

### The Object System
```
Room (contains objects)
  → Furniture (heavy, non-portable, optional surfaces)
    → Bed (has "top" surface for items, "underneath" surface for storage)
    → Wardrobe (has "interior" for clothing)
  → Items (portable objects with weight/size constraints)
  → Doors/Windows (special exits with mutation rules)
```

Each object:
- Inherits from templates (reduces code duplication)
- Tracks location (parent object) and contents (child objects)
- Has keywords for parser matching
- Stores mutations (how it changes when interacted with)
- Weight-checked when moved/picked up

### The Mutation Model
When a player types "BREAK WINDOW":
1. Parser matches verb BREAK + object window
2. Engine looks up window.mutations.break
3. Engine applies the mutation:
   - window becomes "broken" (window.broken = true)
   - window.type changes from "window" to "hole in wall"
   - New description: "Jagged shards of leaded glass cling to the stone frame..."
   - Spawns glass-shard objects
   - Message shown to player
4. **The code IS the state** — no separate flags needed

## 📰 OP-ED: "WHY CODE MUTATION IS THE FUTURE OF INTERACTIVE FICTION"
### *By Frink, Chief Researcher*

For 40 years, interactive fiction has treated state as a **secondary concern**. We craft beautiful prose, define complex verb systems, build elegant parsers — and then store player actions in a dusty corner as flags: `hasUmbrellaFlag = true`, `mirrorBroken = 0|1`, `bedroomVisited++`.

This is architectural malpractice.

The MMO project has committed to a different philosophy: **Code IS state. Code IS data. The two are not separate concerns.**

Let me explain why this matters.

### The Traditional Approach (Is Broken)

In a classic text adventure, here's what happens when the player breaks the mirror:

```lua
-- The old way
if command == "BREAK MIRROR" then
  mirrorBroken = true
  descriptionFlag = "shattered"
  spawnGlassShards()
  print("The mirror shatters with a loud CRACK!")
end
```

You've now created a split reality:
- The **code logic** (if-statements, conditions)
- The **state storage** (flags, database rows)

They're divorced. When the mirror is broken, you have to remember to:
- Update the flag
- Update the description
- Update the collision system
- Update the NPC dialogue
- Update the item list

Forget one? Your game is inconsistent. The mirror looks broken but the player can still use it. The game *lies*.

### The Mutation Model (Is Elegant)

Here's how we do it in the MMO:

```lua
mirror = {
  name = "mirror",
  broken = false,
  mutations = {
    break = function(self)
      self.broken = true
      self.description = "Jagged shards of leaded glass cling to the stone frame."
      spawnObjects({"glass_shard", "glass_shard", "glass_shard"})
    end
  }
}
```

**The code IS the state.** When `mirror.broken` is true, the mirror behaves differently. The parser respects it. The collision system sees it. The description reflects it. One source of truth.

More importantly: **The code is self-modifying.** Lua allows us to write game logic that rewrites itself at runtime. A magical artifact can change how verbs work. An NPC can modify their own dialogue tree based on player choices. A room can restructure its contents dynamically.

This is *homoiconic programming* — where code and data are interchangeable. Lisp had it in 1958. Smalltalk proved its value. Lua gives us the elegance without the performance penalty.

### Why This Matters for MMOs

In a traditional MMO, persistence is about *storing state* — snapshots of database tables. Event sourcing becomes a nightmare. Universe merging requires complex synchronization.

In our model, persistence is about *storing code*. Each player's universe is a Lua environment with mutations applied to the baseline. Two players merging their universes? It's a code merge, not a database reconciliation. Conflicts are version-control problems, not data-synchronization problems.

Emergent behavior becomes possible. A player's actions literally modify the code that governs their world. Other players see those modifications when they enter that universe. Reality is consistent because **there is no state-code split**.

### The Trade-off

Yes, this is unconventional. Yes, it requires comfort with the idea that "gameplay state" is "executable code." Yes, it means abandoning the familiar patterns of traditional relational databases.

But the payoff is worth it:

1. **Consistency.** No divergence between behavior and state.
2. **Emergent gameplay.** Players can script their own modifications.
3. **Scalability.** Storing code is cheaper than storing state.
4. **Simplicity.** One mental model instead of two.

### The Bet

We're betting that 40 years of "state as a flag" is the wrong architectural choice. We're betting that interactive fiction built on **code-as-state** and **homoiconic mutation** will be fundamentally more fun, more consistent, and more scalable.

In a month, we'll know if we're right.

Until then, I'm breaking mirrors.
