## 🎮 SECTION: BUG BURNDOWN — SEARCH/CONTAINER CLUSTER DEMOLISHED

### Wayne's Brave Browser Play-Test: 5 Bugs in One Session

Wayne logged into the Brave browser and played for 20 minutes. That's all it took. He found 5 nested-object visibility bugs that the desktop test suite had missed:

| Bug | Problem | Impact |
|-----|---------|--------|
| **#85** | Search doesn't traverse `nested` key | Hidden objects stay hidden forever |
| **#86** | Wear auto-pickup from containers | Items disappear when worn from inventory |
| **#88** | Drawer resolver mapped to nightstand parent | Wrong container returns results |
| **#89** | "What's inside?" shows room description | Confused player facing furniture in dark |
| **(related)** | Multiple visibility state issues | Sight/search logic corrupted across clusters |

**The Root Cause:** The recent nested-object refactor was sound, but the search layer hadn't been updated to traverse all nested keys. Visibility gates were applying at the wrong level. The parser was checking "is the drawer visible" before checking "is the drawer's parent visible" — which is backwards.

### Smithers' Surgical Response: 8 Tests, Two Deploys

Smithers analyzed the 5 bugs and realized they traced back to a single architectural miss: **the search system wasn't aware of nested object hierarchies**.

**First Deploy (15 Fixes for #85 + #86):**

```lua
-- BEFORE: search only checked direct inventory
search_results = {}
for item in inventory do
    if item:matches_query(query) then
        table.insert(search_results, item)
    end
end

-- AFTER: traverse nested key for composed objects
local function search_tree(container, query, results)
    for item in container:items() do
        if item:matches_query(query) then
            table.insert(results, item)
        end
        if item.nested then
            search_tree(item.nested, query, results)  -- recurse into nested
        end
    end
end
```

**15 regression tests added** — one for each variation of nested-search edge case (item in container, nested in nested, visibility gate at each level, etc.).

**Deploy Result:** #85 ✅ (search now traverses nested) + #86 ✅ (wear pickup respects container hierarchy)

### Nelson's Verification: All Fixes Confirmed

Nelson ran the regression suite post-deploy:

**Results:** 74/74 test files pass, 0 new failures, all visibility gates working correctly.

### Smithers Continues: #88 + #89 Fixed, 8 More Tests

While Nelson was verifying, Smithers moved on to the drawer/nightstand bug and the "what's inside?" confusion:

**The Drawer Bug (#88):**

Drawers are nested objects inside nightstands. When a player searched for a drawer, the system was returning the nightstand's contents (the drawer's parent) instead of the drawer's contents. Root cause: the resolver was checking `parent[query]` instead of `self[query]`.

```lua
-- BEFORE: drawer resolver mapped to parent
local function resolve_drawer_contents(query)
    return parent:search(query)  -- WRONG: searches nightstand's inventory
end

-- AFTER: drawer resolver checks itself first, then parent
local function resolve_drawer_contents(query)
    local result = self:search(query)
    if not result then
        result = parent:search(query)
    end
    return result
end
```

**The What's Inside Bug (#89):**

A player in a dark room tried `what's inside the nightstand?`. The system was returning the full room description instead of the nightstand's inventory. This happened because the "what's inside" verb was checking room visibility instead of container visibility.

**Fix:** Route "what's inside" through the container system, not the room narration system:

```lua
-- BEFORE: what's_inside routed through room narration
narrate("what's inside", player:in_room():describe())

-- AFTER: what's_inside queries container directly
local container = player:find_container(target)
narrate(container:contents_description())
```

**8 regression tests added** — edge cases for dark-room visibility, nested container queries, parent hierarchy checks.

**Deploy Result:** #88 ✅ + #89 ✅ + all visibility logic cleaned

### Mirror Perfection: #52 Commit Fixes Six Edge Cases in One

CBG had designed the mirror system to show appearance including all worn items, all injuries, all body states. But there were edge cases:

| Mirror Bug | Problem | Root Cause |
|-----------|---------|-----------|
| **#90** | Mirror shows only base appearance, not worn items | Worn-item layer wasn't composited into mirror output |
| **#91** | Double periods in narration | Punctuation not deduplicated in appearance+injury combo |
| **#92** | Injury adjectives missing severity tier | Formatter wasn't reading injury modifier table |
| **#93** | Grammar corruption (noun-adjective agreement) | Injury descriptions not gender-aligned to body parts |
| **#94** | Missing injury visibility state | Injuries that shouldn't be visible (internal) were showing |
| **#95** | Mirror narration included room state | Mirror was pulling ambient description instead of appearance-only |

**Smithers' Fix — Single Commit:**

```lua
-- BEFORE: mirror showed only base appearance
local function mirror_describe(player)
    return player:base_appearance()
end

-- AFTER: mirror shows full composition (worn + injuries + state)
local function mirror_describe(player)
    local desc = player:base_appearance()
    
    -- Layer 1: worn items
    for item in player:worn_items() do
        desc = desc .. " " .. item:appearance_in_mirror()
    end
    
    -- Layer 2: visible injuries with severity
    for injury in player:visible_injuries() do
        local severity = injury:severity_adjective()  -- mild, moderate, severe
        desc = desc .. ", " .. severity .. " " .. injury:name()
    end
    
    -- Layer 3: deduplicate punctuation
    desc = desc:gsub("%.+", ".")  -- multiple periods → single
    
    -- Layer 4: verify gender agreement
    desc = verify_grammar(desc, player:pronouns())
    
    return desc
end
```

## 🏗️ SECTION: ARCHITECTURE & DESIGN DECISIONS

### Bart: Material Consistency Added as Core Principle 9

Bart added a new architecture principle to `docs/architecture/objects/core-principles.md`:

**Core Principle 9: Material Consistency**

> Objects must honor their material properties. A wooden door responds to fire differently than a steel door. A glass bottle breaks into glass shards, not plastic. Material drives animation, durability, sound, and interaction rules. **Exception:** Instance-level overrides allowed for special cases (e.g., "This magic door is wooden but unburnable").

This principle closes a design gap from earlier. Objects need to behave consistently based on what they're *made of*, not just what category they're in. It also formalizes the instance-override pattern that came up during puzzle design.

### CBG: Chest Object Fully Designed

CBG designed the chest object (`docs/objects/chest.md`) — a two-handed container with open/close state machine:

```
Chest Specification:
├── Physical Properties
│   ├── Material: oak wood (burnable, not shatterable)
│   ├── Size: two-handed (requires both hands to carry)
│   ├── Weight: 40 kg (heavy, slow pickup)
│   └── Durability: 80 HP (survives moderate damage)
├── Container Properties
│   ├── Capacity: 8 items
│   ├── Visibility: contents hidden when closed
│   └── Access: drop/take locked when closed
├── State Machine
│   ├── locked → (unlock with key) → closed
│   ├── closed → (open verb) → open
│   ├── open → (close verb) → closed
│   └── any state → (break verb) → destroyed
└── Interaction Chain
    ├── examine → see decorative hinges, ornate lock
    ├── open → "You open the oak chest with effort." (narrative beat)
    ├── look inside → list contents with detail
    └── take from closed → "The chest is closed." (access denied)
```

This design will ship in Phase 6. It's the first two-handed container, which introduces new weight/encumbrance mechanics.
