## 🎮 WAVE 2: THE POISON BOTTLE & BEAR TRAP DESIGN SPRINT

### Wayne's Questions Change Everything

Wayne looked at the morning's clean test suite and asked the question that launched the rest of the session:

> "What are the .lua to engine hooks? Is there an engine event? Do we need one?"

This wasn't a feature request. It was an architectural probe. Wayne saw that objects (poison bottles, bear traps) needed to *cause effects* (injuries, state changes, death) — but there was no unified mechanism for an object to tell the engine "this happened to the player." Each object was ad-hoc: the taste handler for poison called `os.exit(0)` directly. The bear trap would need its own death logic. Every new hazardous object would reinvent the wheel.

Wayne's question was really: *"Where is the effect processing pipeline?"*

The answer was: it didn't exist yet.

### CBG: Deep Design Analysis

CBG (Design Lead) produced two comprehensive design documents that defined the session's ambition:

**Poison Bottle Design — 710 lines:**

```
POISON BOTTLE ARCHITECTURE:
├── Container: glass bottle (breakable, transparent)
│   └── Contents: liquid (poison type determines effects)
├── Nested Parts: cork, label, liquid
├── Interaction Chain:
│   ├── examine → see label (fair warning #1)
│   ├── read label → explicit warning text (fair warning #2)
│   ├── smell → acrid/sweet/bitter (fair warning #3)
│   ├── taste → CONSUMPTION PIPELINE BEGINS
│   └── drink → FULL DOSE (accelerated pipeline)
└── Consumption → Injury Pipeline:
    ├── Type A (Hemlock): slow paralysis, 15-turn death
    ├── Type B (Arsenic): nausea → weakness → death (25 turns)
    └── Type C (Nightshade): hallucination → unconsciousness → death
```

**The Fair Warning Chain** is the design centerpiece. The game gives the player *three chances* to realize this is poison before they drink it. Labels, smell, taste — each progressively more explicit. Only a player who ignores all three warnings actually gets poisoned. This is classic interactive fiction design: deadly objects are fair if the game communicates danger clearly.

**Bear Trap Design — 972 lines:**

```
BEAR TRAP FINITE STATE MACHINE:
    [set/armed] ──(touch)──→ [triggered/closed]
         │                         │
    (disarm attempt)          (pry open)
         │                         │
    [disarmed/safe]          [open/resettable]
         │                         │
    (pick up)               (re-arm)
         │                         │
    [inventory item]         [set/armed]

CONTACT INJURY MODEL:
    touch → trap triggers → crushing injury → FSM state change
    Severity: based on body part + trap strength
    Disarm: requires tool (stick, sword) OR high skill check
```

The bear trap introduced a new concept: **touch-triggered effects**. Unlike the poison bottle (which requires deliberate consumption), the bear trap fires on *any contact*. Step on it, reach into it, poke it with your finger — trigger. This required a fundamentally different event model.

### Bart: The Architecture Audit

Bart (Engine Lead) audited the existing engine hooks and found the problem Wayne had intuited:

**Finding:** The injury bridge between objects and the engine was completely ad-hoc. Each object that could hurt the player had its own inline code for damage, state changes, and death checks. There was no common pipeline.

**Worst Offender:**

```lua
-- Found in taste handler (actual current code)
if object.poison then
    player.health = 0
    os.exit(0)  -- ← THIS. The taste handler calls os.exit(0).
end
```

The taste handler for poison literally *exits the Lua process*. No death message. No save. No injury system involvement. Just... crash to terminal. This worked for testing but was architecturally catastrophic.

**Bart's Proposal:** A unified **Effect Processing Pipeline** — a single module that any object can call to apply effects to any entity. Objects declare *what happened* ("player consumed poison type A"), and the pipeline handles *what that means* (injury creation, state changes, narration, death checks).

### Frink: The Research

Frink (Research Specialist) produced a **24KB research document** covering poison and trap mechanics across classic interactive fiction:

| Source | Poison Mechanic | Trap Mechanic |
|--------|----------------|---------------|
| **Zork** | Instant death, minimal warning | Pressure-plate, no escape once triggered |
| **Inform 7** | Timed poison with antidote mechanic | State-machine traps with disarm verbs |
| **TADS** | Progressive symptom chain | Contact-trigger with skill-based disarm |
| **MUDs** | DoT (damage over time) with resist checks | Area-effect traps, party-wide damage |

**Key Insight from Research:** Every successful poison system uses *progressive symptoms* (not instant death), and every successful trap system uses a *finite state machine* (not a boolean flag). The MMO's design aligned with both best practices.

### EP1 🏗️ — Architecture (Bart)

Bart authored the architecture document for the unified Effect Processing Pipeline. The design:

```lua
-- Effect Processing Pipeline (EP1 Architecture)
--
-- Any object can emit an effect:
--   effects.process(source, target, effect_type, params)
--
-- The pipeline:
--   1. NORMALIZE  — canonical form for the effect
--   2. INTERCEPT  — registered interceptors can modify/block
--   3. APPLY      — built-in handlers execute the effect
--   4. NOTIFY     — observers get informed (narration, UI, logs)

-- Built-in effect handlers:
local handlers = {
    damage    = function(target, params) ... end,  -- HP reduction
    injury    = function(target, params) ... end,  -- injury creation
    state     = function(target, params) ... end,  -- state change (consciousness, poison)
    death     = function(target, params) ... end,  -- death processing
    narration = function(target, params) ... end,  -- player-facing text
}
```

**The `os.exit(0)` Discovery:** During architecture work, Bart confirmed that the taste handler's `os.exit(0)` call was the only death pathway for poison. The new pipeline would replace this with a proper `death` effect handler that processes narration, saves state, and ends the game gracefully.

### EP3 ⚛️ — Implementation (Smithers)

Smithers built `src/engine/effects.lua` — the new unified effect processing module:

```lua
-- src/engine/effects.lua — Unified Effect Processing Pipeline
local effects = {}

-- Core pipeline entry point
function effects.process(source, target, effect_type, params)
    local effect = effects.normalize(effect_type, params)
    
    -- Interceptor chain (can modify or block effects)
    for _, interceptor in ipairs(effects._interceptors) do
        effect = interceptor(source, target, effect)
        if not effect then return nil end  -- blocked
    end
    
    -- Find and execute handler
    local handler = effects._handlers[effect.type]
    if handler then
        return handler(source, target, effect)
    end
    
    return nil  -- unknown effect type
end

-- Register custom effect handlers
function effects.register(effect_type, handler_fn)
    effects._handlers[effect_type] = handler_fn
end

-- Register interceptors (armor, resistance, immunity)
function effects.add_interceptor(interceptor_fn)
    table.insert(effects._interceptors, interceptor_fn)
end

-- Normalize effect to canonical form
function effects.normalize(effect_type, params)
    return {
        type = effect_type,
        severity = params.severity or 1,
        duration = params.duration or 0,
        source_type = params.source_type or "unknown",
        narration = params.narration or nil,
        -- ... additional normalized fields
    }
end

-- 5 built-in handlers
effects._handlers = {
    damage    = require("engine.effects.damage"),
    injury    = require("engine.effects.injury"),
    state     = require("engine.effects.state"),
    death     = require("engine.effects.death"),
    narration = require("engine.effects.narration"),
}
```

**Test Results After EP3:** 1,361/1,362 tests pass. The 1 failure was **pre-existing** — not introduced by the pipeline. Zero regressions.

**Why This Matters:** The pipeline is *extensible*. Future objects (acid pools, fire traps, cursed weapons) register their effect types and handlers. The engine doesn't need to know about every possible harm — it just runs the pipeline.

## 📐 THE ARCHITECTURE: WHY THE EFFECTS PIPELINE MATTERS

### Before: The Spaghetti

```
BEFORE THE PIPELINE:
    poison_bottle.lua ──→ os.exit(0)           ← death!
    hit_verb.lua      ──→ inline injury code   ← custom!
    sleep_fix.lua     ──→ inline tick code     ← different!
    bear_trap.lua     ──→ ???                  ← not built yet
    acid_pool.lua     ──→ ???                  ← not built yet
    cursed_weapon.lua ──→ ???                  ← not built yet
    
    Every object reinvents: damage, injury, state change, death, narration
```

### After: The Pipeline

```
AFTER THE PIPELINE:
    poison_bottle.lua ──┐
    hit_verb.lua      ──┤
    sleep_fix.lua     ──┤──→ effects.process() ──→ normalize
    bear_trap.lua     ──┤                           ├── intercept
    acid_pool.lua     ──┤                           ├── apply
    cursed_weapon.lua ──┘                           └── notify
    
    Every object calls ONE function. Pipeline handles everything.
```

### The Interceptor Pattern

The pipeline's interceptor chain is where future gameplay gets interesting:

```lua
-- Armor interceptor (reduces damage effects)
effects.add_interceptor(function(source, target, effect)
    if effect.type == "damage" or effect.type == "crush" then
        local armor = target:get_armor(effect.body_part)
        if armor then
            effect.severity = effect.severity * (1 - armor.reduction)
            if effect.severity <= 0 then
                return nil  -- armor fully blocked the effect
            end
        end
    end
    return effect
end)

-- Poison resistance interceptor (future: potions, racial traits)
effects.add_interceptor(function(source, target, effect)
    if effect.type == "poison" and target:has_trait("poison_resistant") then
        effect.severity = effect.severity * 0.5
        effect.narration = "You feel queasy, but your constitution fights it off."
    end
    return effect
end)
```

This is how the MMO scales. New objects don't need new engine code. New defenses don't need per-object checks. The pipeline is the connective tissue between "things that cause effects" and "things that modify or respond to effects."
