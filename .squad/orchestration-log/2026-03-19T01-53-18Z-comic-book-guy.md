## Design Patterns Established

1. **Tool Convention (requires_tool / provides_tool)** — Objects declare capabilities they provide; mutations declare capabilities they require. Engine resolves via capability matching (any fire source lights any candle). Distinct from key pattern (specific item-ID matching).

2. **Consumable Charges** — Tools have limited uses. Each use decrements via D-14 code rewrite. When depleted, mutates to `when_depleted` variant.

3. **Composed Tool Messages** — Tool's `use_message` + target's `message` + optional `depleted_message` on depletion.

4. **First Puzzle (Light the Candle)** — Player wakes in dark → opens nightstand drawer → finds matchbox → takes candle → "light candle" → room illuminated. Two-step discovery teaches containment system. Anti-softlock: door open, daytime window light, 3 matches margin for error.

## Key Design Decisions

- **Capability matching extensible** — Tinderbox and matchbox both provide `fire_source`
- **3 charges is Goldilocks number** — One is cruel (no margin), five is generous (no tension), three is right
- **Co-locate solution components** — Candle on top, matches in drawer; no cross-room hunting
- **Mutation handles tool depletion** — matchbox → matchbox-empty is same pattern as candle → candle-lit
