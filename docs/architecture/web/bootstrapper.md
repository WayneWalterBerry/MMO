# Bootstrapper — Layer 1 JavaScript Loader

**Author:** Bart (Architect)  
**Date:** 2026-07-22  
**Status:** Design

## Purpose

The bootstrapper is a small JavaScript file (~5-10KB) that serves as the **single entry point** for the web game. It is the ONLY JavaScript file that `index.html` loads directly. Its job:

1. Show status messages as things load
2. Fetch the compressed engine bundle
3. Decompress it
4. Feed the decompressed Lua code to Fengari
5. Fetch the game adapter (coroutine bridge)
6. Hand control to the Lua engine

The bootstrapper contains **zero game logic**. It is purely a delivery mechanism.

---

## File

```
/play/bootstrapper.js   (~5-10KB)
```

Loaded by `index.html` via a single `<script>` tag:

```html
<!-- Fengari loaded from CDN first -->
<script src="https://cdn.jsdelivr.net/npm/fengari-web@0.1.4/dist/fengari-web.js"></script>
<!-- The ONLY local JS file -->
<script src="bootstrapper.js"></script>
```

## Status Messages

The bootstrapper writes light gray status text directly to the terminal output element as it progresses. These messages use the same DOM element as the game output — they appear inline, not as a separate splash screen.

**Color:** Light gray (`#888` or similar — distinct from game text but not distracting)

```
Loading Bootstrapper...         ← immediately on script execution
Loading Game Engine...          ← before fetch() starts
Decompressing Engine...         ← after fetch completes, before decompression
Initializing Game Engine...     ← after decompression, before Fengari execution
```

### Implementation

```javascript
const outputEl = document.getElementById('output');

function showStatus(message) {
    const line = document.createElement('div');
    line.textContent = message;
    line.style.color = '#888';
    outputEl.appendChild(line);
}
```

Once the Lua engine takes over, subsequent status messages ("Loading Room: Bedroom...", etc.) are generated by the JIT loader in Lua, using the same display output mechanism.

## Execution Flow

### Step 1: Initialize

```javascript
showStatus('Loading Bootstrapper...');
```

The bootstrapper announces itself. At this point, Fengari is already loaded (from the CDN `<script>` tag that precedes bootstrapper.js in the HTML).

### Step 2: Fetch Compressed Engine

```javascript
showStatus('Loading Game Engine...');

const response = await fetch('/play/engine.lua.gz');
const compressedData = await response.arrayBuffer();
```

A single `fetch()` call retrieves the pre-compressed engine bundle (~500KB). This is the main download — everything else is small.

### Step 3: Decompress

```javascript
showStatus('Decompressing Engine...');

const engineSource = await decompress(compressedData);
```

Decompression uses the browser-native `DecompressionStream` API:

```javascript
async function decompress(compressedBuffer) {
    const ds = new DecompressionStream('gzip');
    const writer = ds.writable.getWriter();
    const reader = ds.readable.getReader();

    writer.write(compressedBuffer);
    writer.close();

    const chunks = [];
    while (true) {
        const { done, value } = await reader.read();
        if (done) break;
        chunks.push(value);
    }

    const decoder = new TextDecoder();
    return chunks.map(c => decoder.decode(c, { stream: true })).join('');
}
```

#### Fallback for Older Browsers

If `DecompressionStream` is not available, fall back to a small JS decompression library (e.g., [fflate](https://github.com/101arrowz/fflate) at ~3KB minified):

```javascript
async function decompress(compressedBuffer) {
    if (typeof DecompressionStream !== 'undefined') {
        return decompressNative(compressedBuffer);
    }
    // Fallback: use fflate (bundled inline in bootstrapper.js)
    const decompressed = fflate.decompressSync(new Uint8Array(compressedBuffer));
    return new TextDecoder().decode(decompressed);
}
```

`DecompressionStream` is supported in Chrome 80+, Firefox 113+, Safari 16.4+, Edge 80+. The fallback handles the long tail.

### Step 4: Load Engine into Fengari

```javascript
showStatus('Initializing Game Engine...');

// Execute the engine Lua source in Fengari
fengari.load(engineSource, 'engine')();
```

After this call, all engine modules are available via `require()` inside the Fengari Lua environment.

### Step 5: Fetch Game Adapter

```javascript
const adapterResponse = await fetch('/play/game-adapter.lua');
const adapterSource = await adapterResponse.text();

// Execute the adapter — this wires up the coroutine bridge
fengari.load(adapterSource, 'game-adapter')();
```

The game adapter sets up:
- VFS for engine module resolution
- `io.read()` override (yields coroutine, JS resumes with player input)
- `print()` override (routes to DOM output element)
- The main game coroutine

### Step 6: Hand Off to Lua

After the adapter executes, control passes to the Lua engine. The bootstrapper's job is done. All subsequent loading (templates, levels, rooms, objects) is handled by the JIT loader in Lua.

The bootstrapper remains in memory only to support the coroutine resume mechanism (JS calls back into the Lua coroutine when the player types a command).

## Size Budget

| Component | Size |
|-----------|------|
| Core bootstrapper logic | ~2KB |
| DecompressionStream wrapper | ~1KB |
| fflate fallback (optional) | ~3KB |
| Status message UI | ~0.5KB |
| **Total** | **~5-7KB** |

This is intentionally tiny. The bootstrapper should load and execute in under 50ms on any connection.

---

## Error Handling

| Failure | Behavior |
|---------|----------|
| Fengari CDN fails | Fatal — show "Failed to load game runtime" error. Nothing works without Fengari. |
| engine.lua.gz fetch fails | Retry once after 2s. If still fails, show "Failed to load game engine" error. |
| Decompression fails | Try fallback (fflate). If fallback also fails, show "Failed to decompress game engine" error. |
| game-adapter.lua fetch fails | Retry once after 2s. If still fails, show "Failed to load game adapter" error. |
| Fengari execution error | Log full error to console. Show "Game engine initialization failed" error. |

Error messages appear in the same output area as status messages, but in red (`#cc4444`) instead of gray.

## Relationship to Existing Code

### What It Replaces

The current `index.html` loads three things:
1. Fengari from CDN
2. `game-bundle.js` (16MB monolith — ALL Lua as JS strings)
3. `game-adapter.lua` (via Fengari)

The bootstrapper replaces `game-bundle.js` with a fetch+decompress+load pattern for `engine.lua.gz`. The adapter fetch moves inside the bootstrapper (no longer a separate Fengari `<script>` tag).

### What It Doesn't Change

- Fengari is still loaded from CDN (same `<script>` tag)
- The game adapter's Lua code is unchanged — only HOW it's loaded changes
- The coroutine bridge mechanism is preserved
- All engine and game code remains pure Lua

---

## Security Considerations

- `engine.lua.gz` is fetched from the same origin (GitHub Pages). No CORS issues.
- Lua code is executed via Fengari's sandboxed environment — no `eval()` on arbitrary JS.
- The compressed bundle is a known artifact — not user-supplied content.
- `DecompressionStream` operates on binary data, not executable code. The decompressed result is a Lua string, not JavaScript.

---

## Implementation Notes for Smithers

1. **Start with the happy path** — get fetch → decompress → Fengari load working first. Add fallbacks and error handling second.
2. **Test decompression separately** — compress a small Lua file with gzip, verify round-trip through DecompressionStream.
3. **The bootstrapper replaces `game-bundle.js`** — don't try to make them coexist. Remove the old `<script src="game-bundle.js">` tag.
4. **Status messages are critical UX** — without them, the player stares at a blank screen for 1-2 seconds. Wire them up early.
5. **The adapter fetch can happen in parallel** with engine decompression — use `Promise.all()` if desired, but sequential is fine for V1.
